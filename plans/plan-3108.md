# Issue #3108 — Research storage abstraction & `research:` profile block

## Summary

Lay the foundation for the research-augmented planning cycle (Epic [#3107](plans/plan-3107.md)) by (a) adding a typed `research:` block to [.polyphony-config/profile.yaml](.polyphony-config/profile.yaml) with first-class loader + validator support, and (b) introducing a cross-platform research-storage abstraction that mirrors polyphony's existing paired-client pattern (GitHub via [src/Polyphony/Infrastructure/Processes/IGhClient.cs](src/Polyphony/Infrastructure/Processes/IGhClient.cs), ADO via [src/Polyphony/Infrastructure/AzureDevOps/IAdoClient.cs](src/Polyphony/Infrastructure/AzureDevOps/IAdoClient.cs)). No agent, sub-workflow, or curation logic — those land in Issues 2–4.

## Problem / Motivation

Today, [.polyphony-config/profile.yaml](.polyphony-config/profile.yaml) is only checked for existence (`V-14` warning in [src/Polyphony/Configuration/ConfigValidator.cs](src/Polyphony/Configuration/ConfigValidator.cs); see lines 240–245). Its contents are not parsed, typed, or validated. The Epic requires architects to declare `research_needs` and have results synthesized into a sibling research repo, with opt-in driven purely by adding a `research:` block — no code changes per consuming repo. Without a typed profile block and a platform-agnostic storage seam, every downstream Issue (assistant, escalation, curation, integration) would either re-invent its own config parsing or hard-bind to one platform. Both are P5/P8 violations: process knowledge belongs in config, and the validator must remain the oracle for legality.

## Proposed Approach

**Profile-block schema** — Extend the profile-loading pipeline alongside [src/Polyphony/Configuration/ProcessConfig.cs](src/Polyphony/Configuration/ProcessConfig.cs) and the family in [src/Polyphony/Configuration/](src/Polyphony/Configuration/). Add a typed `ProfileConfig` (or extend whatever already deserializes profile.yaml — currently only existence-checked) with a nested `ResearchConfig` record carrying:

  - `repo` (string, `owner/name` or ADO `org/project/repo`) — required when block present
  - `branch` (string, default `main`) — optional
  - `platform` (enum: `github` | `ado`) — required; drives storage-impl selection
  - `auth` (sub-record: `kind: env|gh-cli|ado-cli`, `env_var` when applicable) — required

Keep the schema minimal but extensible: deserialize unknown keys into a forward-compat bag rather than rejecting (consistent with YamlDotNet's lenient default).

**Validator rules** — Add new rule codes to [src/Polyphony/Configuration/ConfigValidator.cs](src/Polyphony/Configuration/ConfigValidator.cs) following the existing `V-N` pattern (next free codes after the V-1–V-21 family; V-17/V-18 are retired and may be reused or skipped — see open question). Rules:

  - **error**: `research:` present but missing `repo` / `platform` / `auth.kind`
  - **error**: `platform` not one of the supported values
  - **error**: `auth.kind: env` without `env_var`
  - **warning**: `research:` absent (research features will be disabled — informational, not a failure)

Messages must be actionable per polyphony's validator-as-oracle principle (P8): say what's wrong, where, and how to fix it.

**Cross-platform storage abstraction** — Polyphony's actual "platform-router pattern" is the paired-client convention: [IGhClient](src/Polyphony/Infrastructure/Processes/IGhClient.cs) and [IAdoClient](src/Polyphony/Infrastructure/AzureDevOps/IAdoClient.cs) implement parallel verbs, with the verb layer (e.g., [PrCommands.MergePlanPr.cs](src/Polyphony/Commands/PrCommands.MergePlanPr.cs) vs [PrCommands.MergePlanAdo.cs](src/Polyphony/Commands/PrCommands.MergePlanAdo.cs)) selecting based on context. Reuse this pattern: introduce `IResearchStorage` with two narrow, append-friendly operations sufficient for Issues 2–6 (`WriteFileAsync(path, content, commitMessage)` and `ReadFileAsync(path)` against the configured `repo`/`branch`); ship `GitHubResearchStorage` (delegates to `IGhClient` / `gh api`) and `AdoResearchStorage` (delegates to `IAdoClient` Git Items API). A factory selects the impl from `ResearchConfig.Platform`. No new platform abstraction layer — strictly the existing paired-client convention.

**Documentation** — Update [docs/polyphony-process-config-schema.md](docs/polyphony-process-config-schema.md) (or the profile-schema doc if separate) with the `research:` block reference and a worked example for each platform.

## Acceptance Criteria

- [ ] `research:` block schema is defined as a typed C# model under `src/Polyphony/Configuration/`, deserialized from [.polyphony-config/profile.yaml](.polyphony-config/profile.yaml), and documented in the profile-schema doc.
- [ ] Validator emits actionable errors for missing/malformed `research:` fields with stable rule codes following the existing `V-N` family.
- [ ] `IResearchStorage` exists with `GitHubResearchStorage` and `AdoResearchStorage` impls, selected by config — no per-repo code changes required to switch platforms.
- [ ] Unit tests cover (a) schema deserialization happy path + each malformed shape, (b) validator rules, (c) factory selection per `platform` value (mocked underlying clients — no live API calls).
- [ ] A fresh repo can opt in by adding only the `research:` block — no code changes required (demonstrated via integration-style test or fixture).
- [ ] Build passes with zero errors and warnings (`dotnet build --no-restore`).
- [ ] All existing tests pass; new tests cover changed behavior (`dotnet test --no-restore`).
