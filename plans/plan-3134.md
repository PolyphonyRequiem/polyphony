# Plan — `researcher` sufficiency check + `deep-researcher` escalation (AB#3134)

## Summary
Extend the research sub-workflow delivered by AB#3132 (issue-2) with a two-step escalation pattern: a `researcher` sufficiency-judge agent inspects the archive-only findings produced by `research-assistant`, and — when the archive is judged insufficient — invokes a `deep-researcher` agent whose tool surface includes web / GitHub / MCP. Escalations are capped per `research_needs` topic (default `1`), with the cap overridable via the `research:` block in [.polyphony-config/profile.yaml](.polyphony-config/profile.yaml).

## Problem / Motivation
The archive-only first pass (issue-2) is cheap but blind: when prior art does not exist in the archive, `research-assistant` returns thin or empty findings and the architect has nothing to learn from. The locked three-tier pattern in [plans/plan-3131.md](plans/plan-3131.md) calls for a sufficiency judge that decides when to spend a more expensive escalation, plus a deep tier that can actually reach live sources. Without this issue, the sub-workflow is functionally archive-only and the Epic's success criterion ("findings cite both archive and live sources when needed") cannot be met.

## Proposed Approach
All changes live inside the research sub-workflow file and adjacent agent definitions established by issue-2 — no architect-side wiring (that's issue-5).

1. **Add `researcher` agent** — sufficiency judge. Input: the `research_needs` topic + the `research-assistant` findings document. Output: structured verdict `{ sufficient: bool, gaps: [...], rationale: string }`. No tools beyond reading the prior step's output.
2. **Add escalation route** — when `sufficient = false` AND the topic's escalation count < cap, route to `deep-researcher`; otherwise terminate the loop and pass the archive-only findings forward.
3. **Add `deep-researcher` agent** — extended tools (web / GitHub / MCP). Input: `research_needs` topic, archive findings, and the `gaps` list from the judge. Output: an augmented findings document that merges archive citations with live-source citations (URL + capture date — the rigor bar lives in issue-6, but deep-tier output must already carry both kinds of citations).
4. **Extend `research:` profile schema** — add an optional `escalation_cap_per_topic` field (default `1`). Validation rejects negative values; `0` disables escalation entirely (archive-only mode).
5. **Per-topic counter** — track escalation count keyed by `research_needs` topic id within a single sub-workflow invocation, used by the route condition in step 2.
6. **Tests** — extend the harness scenarios from issue-2:
   - archive-sufficient case: judge returns `sufficient=true`, no deep call, single-source citations.
   - archive-insufficient case under cap: escalates exactly once, output cites both archive and live sources.
   - cap-exhausted case: second insufficient verdict does NOT escalate; archive-only findings are returned with a recorded "cap reached" marker.
   - `escalation_cap_per_topic: 0`: never escalates, judge step still runs.

## Acceptance Criteria
- [ ] Archive-insufficient cases escalate to the `deep-researcher` tier with extended tools (web / GitHub / MCP).
- [ ] Escalation cap defaults to `1` per `research_needs` topic and is overridable via the `research:` profile block.
- [ ] Findings cite both archive and live sources when the deep tier is invoked.
- [ ] Harness scenarios cover sufficient / insufficient-under-cap / cap-exhausted / cap-zero paths.
- [ ] Build passes with zero errors and warnings.
- [ ] All existing tests pass; new tests cover changed behavior.

## Dependencies
- AB#3132 (issue-2) — research sub-workflow + `research-assistant` archive-only first pass. The `research_needs` payload schema and the findings-document schema defined there are consumed unchanged.
