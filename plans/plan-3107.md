# Research-augmented planning cycle (Epic #3107)

## Strategic Objective

Close polyphony's planning loop by giving any architect — at any planning depth — a sub-workflow that can pause, learn, and resume with synthesized findings. Knowledge accumulated during a run is curated into a per-repo, Obsidian-friendly research wiki so subsequent planning runs discover prior art instead of re-deriving it. This dogfoods the feature on polyphony itself and exercises the full architect → review → plan loop end-to-end on a non-trivial design problem (post PR #325 / #326 structural fixes).

## Success Criteria

Acceptance criteria are **relaxed** for this first unattended dogfood run (per the Epic body). The Minimum bar is what closes this Epic; Stretch items may defer to follow-on Epics.

### Minimum (must land for Epic close)

- An architect at any planning level can declare structured `research_needs` and receive a synthesized findings document before its next iteration.
- A fresh repo opts in via the `research:` block in [.polyphony-config/profile.yaml](.polyphony-config/profile.yaml) alone — no code changes.
- Same-platform path proven end-to-end (github↔github sufficient).
- Citations include source URL and capture date.
- Research output bypasses the plan-review gate and feeds directly into the requesting architect's next iteration.

### Stretch (defer if needed)

- Cross-platform proven (e.g. ADO source ↔ GitHub research) via the existing platform-router pattern.
- Kept articles discoverable via `INDEX.md` with Johnny-Decimal IDs.
- Scratch under `research/scratch/<run-id>/` pruned cleanly at apex close, end-to-end verified.
- Citations carry a freshness signal beyond URL + capture date.

## Scope

### In Scope

- New `research:` block schema in [.polyphony-config/profile.yaml](.polyphony-config/profile.yaml), with validator coverage.
- Cross-platform research-storage abstraction (sibling repo on GitHub or ADO), reusing polyphony's platform-router pattern.
- Three-tier researcher pipeline: `research-assistant` → `researcher` → `deep-researcher` (escalation only).
- `archivist` curation pass at end of each research run (keep / discard / expand).
- Hybrid storage: scratch on the feature branch under `research/scratch/<run-id>/`; kept articles graduate to the sibling research repo (Obsidian-friendly markdown, Johnny-Decimal layout, cited sources with capture date).
- Architect prompt/output contract additions: emit `research_needs`; loop-back routing in workflows under [.conductor/registry/workflows/](.conductor/registry/workflows/) that bypasses review when research is the next step.
- Scratch pruning at apex close.
- `INDEX.md` maintenance and Obsidian polish.
- End-to-end demonstration on this very Epic's planning run.

### Out of Scope

- Migrating existing in-tree docs into the research wiki.
- Cross-repo research federation (one wiki feeding multiple source repos).
- Human-in-the-loop research authoring UI.
- Any change to the children-json sidecar / from-ref recovery seams shipped in PR #325 / #326 (treated as fixed dependencies).

## Decomposition Approach

The Epic body offers a six-phase decomposition. The phases align cleanly with one merge group apiece (the explicit goal: "exercise the wave logic"), so the plan preserves them as six Issues with the dependency chain implied by the architecture:

- **Issue 1 — Storage + config foundation** is the platform other phases stand on.
- **Issues 2 → 3 → 4** form the research pipeline (assistant → escalation → curation), each layering on the prior tier.
- **Issue 5 — Architect integration** can begin once the assistant tier (Issue 2) returns synthesized output, but its end-to-end demo requires the full pipeline.
- **Issue 6 — Wiki polish** depends on the archivist (Issue 4) being in place to write articles worth indexing.

No Tasks are listed here — Issues will plan their own Task decomposition.

## Child Issues

- task-1 — Research storage abstraction & `research:` profile block — foundation: schema, validator, cross-platform remote abstraction.
- task-2 — Research sub-workflow & `research-assistant` agent — archive-only first pass invocable by any architect.
- task-3 — `researcher` sufficiency check & `deep-researcher` escalation — extended-tool fallback.
- task-4 — `archivist` curation pass — keep/discard/expand with sibling-repo write.
- task-5 — Architect integration: `research_needs` emission & review-bypass loop — wires the subsystem into existing planning workflows.
- task-6 — Wiki polish: Johnny-Decimal numbering, Obsidian compatibility, citation rigor, `INDEX.md` maintenance.

## Risks

- **Recursion trap during planning**: this Epic's own planner must NOT invoke a research sub-workflow (the subsystem doesn't exist yet). Architects on child Issues must be similarly careful until Issue 2 lands.
- **Cross-platform abstraction drift**: the existing platform-router pattern must be reused, not re-invented; deviating is a likely source of cross-platform Stretch slipping further.
- **Review-bypass safety**: routing research output back into the architect without a gate is correct for context, but the loop must terminate (bounded research iterations) to avoid runaway cycles.
- **Dogfood scope creep**: AC are relaxed deliberately; child Issues should resist promoting Stretch items into Minimum.
