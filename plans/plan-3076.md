# Plan â€” Architect integration: `research_needs` + review-bypass loop-back (#3076)

## Summary

Wire a `research_needs` output channel into the existing architect node in [`.conductor/registry/workflows/plan-level.yaml`](.conductor/registry/workflows/plan-level.yaml) and add a routing leg that dispatches the request to the research sub-workflow (delivered by sibling #3073) and re-enters the same architect node via the existing `route-back-to-prior-node` primitive â€” the same shape the workflow already uses for `open_questions_answer_counter â†’ architect`. Findings reach the architect's next iteration without traversing the plan-review leg.

## Problem / Motivation

Today the architect agent in `plan-level.yaml` has exactly two outputs that drive routing â€” `open_questions` (loops back to architect after human answer) and the plan/children pair (proceeds to write_plan â†’ review). It cannot pause to acquire context. The parent epic [#3071](plans/plan-3071.md) introduces a research subsystem; this Issue is the integration seam on the architect side. The sub-workflow itself is owned by sibling #3073; the storage abstraction by #3072. This Issue is purely the architect node's input/output shape and the routing edges.

The parent epic locks two non-negotiables this Issue must honor:

1. **No new re-entry primitive.** Re-use the existing pattern in [`plan-level.yaml`](.conductor/registry/workflows/plan-level.yaml) lines ~694â€“711 where `open_questions_answer_counter` ends with `routes: - to: architect`. The research leg must mirror that shape, not invent a new one.
2. **Bypass review.** Research findings are *context, not commitments* â€” the loop-back must NOT touch `write_plan`, `open_plan_pr`, or `plan_reviewer`.

## Proposed Approach

Four surgical changes, all in already-existing files. No new workflow files; no new agents.

### 1. Architect output schema â€” add `research_needs`

In [`.conductor/registry/workflows/plan-level.yaml`](.conductor/registry/workflows/plan-level.yaml) at the `architect` node (line 531), add a fourth top-level output field alongside `plan` / `children` / `open_questions`:

The exact item shape is the **one open question** below â€” it must align with the input contract of the research sub-workflow that #3073 delivers.

### 2. Routing â€” interpose a research dispatch leg before `open_questions_policy`

The architect's current single route is `- to: open_questions_policy`. Replace with an ordered, mutually-exclusive route list that prefers the research leg when `research_needs` is non-empty:

```yaml
routes:
  - to: research_dispatch
    when: "{{ architect.output.research_needs is defined and architect.output.research_needs | length > 0 }}"
  - to: open_questions_policy
```

Research is preferred over open-questions when both are present because research may *answer* what would otherwise become an open question. (If this ordering proves wrong in dogfood, it's a one-line swap; calling it out so the reviewer sees the choice.)

### 3. New nodes â€” `research_dispatch` + `research_loop_counter`

Mirror the shape of `open_questions_answer_counter` (line 694) â€” a script node that performs the work and ends with `routes: - to: architect`. Two nodes:

- **`research_dispatch`** (`type: script`) â€” invokes the research sub-workflow. Initial implementation calls `conductor run research@polyphony` via the CLI with the `research_needs` payload as input, captures the synthesized findings document (markdown) on stdout, and surfaces it as `research_dispatch.output.findings`. The script wrapper lives at `.conductor/registry/scripts/research-dispatch.ps1` (new) and follows the conventions in the existing scripts (`route-actionable-executor.ps1`, `lifecycle-router.ps1`).
- **`research_loop_counter`** (`type: script`) â€” increments a per-work-item counter file using the same `[System.IO.Path]::GetTempPath()` + `conductor-research-loops-{{ workflow.input.work_item_id }}` pattern as `open_questions_counter`. Caps at a configured maximum (default 3, mirrors `open_questions` policy). On cap, routes to `open_questions_policy` (i.e. proceed normally) instead of looping again â€” fail-honest, don't loop forever.

Both nodes route `- to: architect` on the happy path. The cap branch on `research_loop_counter` falls through to `open_questions_policy` so the architect's other gates still fire.
