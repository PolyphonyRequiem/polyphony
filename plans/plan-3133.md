# Research sub-workflow + `research-assistant` (archive-only first pass)

## Summary
Stand up the first-pass research sub-workflow and the cheap-tier `research-assistant` agent (archive-only — no live tools), and define the `research_needs` payload schema as the stable contract that downstream consumers (Issue #3136 — architect integration) bind to. End-to-end callable from the harness, with a synthesized findings document as output.

## Problem / Motivation
The research-augmented planning cycle (Epic #3131) requires a sub-workflow that the architect leg can hand off to with a structured `research_needs` payload and receive back a synthesized findings document. Without a stable schema and a working skeleton, the downstream architect-integration issue (#3136) has no contract to bind to and no callable target to loop back through. This issue delivers the minimum end-to-end path: schema + skeleton sub-workflow + cheap archive-only agent, all driven by the harness so subsequent issues (#3134 sufficiency check, #3135 archivist write, #3136 architect integration) can layer on a stable foundation.

## Proposed Approach
Three-task slice, ordered by dependency:

1. **Define `research_needs` payload schema.** Author a JSON Schema (or YAML-described shape consistent with the conductor verb-envelope contract) capturing the fields the architect emits and the assistant consumes — minimally: `topics[]` (required questions/areas), `context` (originating work-item id, plan excerpt), `budget_hint` (cheap | extended), `archive_scope` (paths/tags to constrain search). Land the schema under [.polyphony-config/](\.polyphony-config/) or a workflow-adjacent path TBD by the workflow-author skill. This is the contract #3136 binds to.
2. **Research sub-workflow skeleton.** Add a new conductor workflow YAML under [workflows/](workflows/) that accepts a `research_needs` payload, routes to the `research-assistant` agent, and returns a synthesized findings document (Markdown). Wire it so it is invocable from the harness (see [tests/harness/](tests/harness/)) end-to-end with a scripted `FakeProvider`. Follow the polyphony-workflow-author conventions (three-vocabulary rule; shell-out idiom; established helper scripts).
3. **`research-assistant` agent (archive-only, cheap tier).** Author the agent prompt + agent-guidance entry (cheap model tier per [.polyphony-config/profile.yaml](.polyphony-config/profile.yaml)). Sources MUST be strictly archive (no live web/MCP tools wired in this pass). Synthesizes findings into a single Markdown document conforming to a simple, documented shape. Add a harness scenario under [tests/harness/scenarios/](tests/harness/scenarios/) that drives the sub-workflow end-to-end with a scripted provider and asserts the findings document is returned.

All three tasks fit comfortably in a single PG (~2000 LoC ceiling), but are sequenced because the schema is the binding contract and the agent depends on the workflow shell.
