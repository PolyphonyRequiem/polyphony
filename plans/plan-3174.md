# Apex-driver pre-flight tag check for previously-processed work items

## Summary

Extend the apex-driver pre-flight in [scripts/Invoke-PolyphonySdlc.ps1](scripts/Invoke-PolyphonySdlc.ps1) — alongside the existing terminal-state refusal at Phase 2.5 — with a Phase 2.6 tag check that fires when the target work item carries any `polyphony:*` state tag. Tag classification routes through the typed `PolyphonyTag` DU introduced by sibling Issue #3173, so the set of polyphony-owned tags has exactly one definition shared with `polyphony reset`. The driver refuses `-Intent new` on a previously-processed item and points the operator at `polyphony reset --root-id N` or `-Intent resume`; `-Intent resume` and `-Intent replan` proceed unchanged.

## Problem / Motivation

From the parent Epic AB#3165 reproducer: re-running `Invoke-PolyphonySdlc.ps1 -ApexId 3127 -Intent new` against a previously-processed root deterministically loops to `scope_revise_cap_gate` even after aggressive cleanup. One pollution surface is ADO **tags** stamped by the previous run (`polyphony:planned`, `polyphony:root`, `polyphony:facets=…`). When polyphony reads the work item on the next run it sees its own prior output as input — the architect skips re-planning, the impl agent observes "no changes needed", and the empty MG dies in the reviewer.

Issue #3172 already closed the *terminal-state* surface (state=Done/Closed/Removed/Resolved). The tag surface is orthogonal: a work item that an operator manually reopened to `To Do` still carries the previous run's `polyphony:*` tags and will still fall into the same trap. We need a second pre-flight refusal that keys off tags, not state.

## Proposed Approach

Add **Phase 2.6 — previously-processed pre-flight refusal** to [scripts/Invoke-PolyphonySdlc.ps1](scripts/Invoke-PolyphonySdlc.ps1), immediately after the existing Phase 2.5 terminal-state block. Mirror the Phase 2.5 idiom (twig show → JSON parse → enum check → friendly refusal with remediation) so the two checks read consistently.

**Routing rule (chosen and documented in the refusal message):**

| Intent | Tag state | Behaviour |
|---|---|---|
| `new`    | any `polyphony:*` tag present | **REFUSE** — emit warning, point to `polyphony reset --root-id N` (clean re-dispatch) or `-Intent resume` (re-attach) |
| `new`    | no `polyphony:*` tags          | proceed |
| `resume` | any                            | proceed (operator is intentionally re-attaching) |
| `replan` | any                            | proceed (operator is intentionally re-planning) |

Rationale for refuse-rather-than-auto-route: silently rewriting `intent=new` to `intent=resume` is a footgun — `resume` semantics depend on per-root state under `<git-common-dir>/polyphony/N/` that may or may not exist. An explicit refusal with two clearly-named remediations (reset vs resume) keeps the operator in the loop, mirrors the Phase 2.5 terminal-state refusal shape, and matches the existing `-Intent resume refused: outcome=created` precedent at Phase 6.

**Tag detection** consumes the typed `PolyphonyTag` DU shipped by Issue #3173. The driver invokes a small CLI helper (likely `polyphony tags classify` or whatever surface #3173 exposes for the reset verb) — or, if #3173 exposes only an in-process API, the driver shells out to a thin wrapper. The script must NOT do `if ($tags -match 'polyphony:')` ad-hoc string matching; AC#2 makes the typed-DU consumption a hard requirement so there is exactly one definition of "polyphony-owned tag" in the codebase.

**Escape hatch.** Reuse the existing `-SkipStateCheck` switch to bypass both Phase 2.5 and Phase 2.6 (single combined escape hatch — the two checks are conceptually one pre-flight). Document the broadened scope in the parameter help block.

**Pester coverage** in [scripts/Invoke-PolyphonySdlc.Tests.ps1](scripts/Invoke-PolyphonySdlc.Tests.ps1):

1. `intent=new` + tagged work item → refuses with non-zero exit, message contains `polyphony reset --root-id` and `-Intent resume`.
2. `intent=resume` + tagged work item → proceeds past Phase 2.6.
3. `intent=new` + untagged work item → proceeds past Phase 2.6.
4. `intent=replan` + tagged work item → proceeds (regression guard for the routing-table row).
5. `-SkipStateCheck` + `intent=new` + tagged work item → proceeds (escape-hatch covers tag check too).

Mock the twig/polyphony shell-outs the same way the existing tests mock `twig show` for the Phase 2.5 cases.

## Acceptance Criteria

- [ ] A second `Invoke-PolyphonySdlc.ps1 -ApexId N -Intent new` invocation against a work item carrying any `polyphony:*` state tag does not silently re-plan; it refuses with a non-zero exit and a remediation message.
- [ ] Tag detection consumes the `PolyphonyTag` DU from Issue #3173; no ad-hoc `polyphony:` string matching exists in the driver script.
- [ ] Pester coverage in [scripts/Invoke-PolyphonySdlc.Tests.ps1](scripts/Invoke-PolyphonySdlc.Tests.ps1) for: tagged + `intent=new` (refuses), tagged + `intent=resume` (proceeds), untagged + `intent=new` (proceeds), tagged + `intent=replan` (proceeds), tagged + `intent=new` + `-SkipStateCheck` (proceeds).
- [ ] Refusal message points to `polyphony reset --root-id N` for a clean re-dispatch and to `-Intent resume` for re-attach.
- [ ] `-SkipStateCheck` parameter help in [scripts/Invoke-PolyphonySdlc.ps1](scripts/Invoke-PolyphonySdlc.ps1) is updated to note the broadened scope (state + tag pre-flight).
- [ ] Build passes with zero errors and warnings.
- [ ] All existing Pester tests pass; new tests cover the routing-table rows above.
