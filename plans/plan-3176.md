# Idempotency invariant harness scenario (deferred)

## Summary
Add a single new scenario under [tests/harness/scenarios/](tests/harness/scenarios/) that double-dispatches the apex driver on the same root via the harness `FakeProvider` and asserts the second run lands in a known-good terminal state — either the zero-diff short-circuit from sibling #3175 or an artifact set identical to run 1 — and fails loudly if the second run reaches `scope_revise_cap_gate` (the AB#3127 failure mode). This is a regression-prevention net, not new product behavior; sequence it after siblings #3172 / #3174 / #3175 land so the "known-good" terminal it asserts on actually exists.

## Problem / Motivation
AB#3127 surfaced that re-running polyphony on the same apex root drives the workflow into `scope_revise_cap_gate` instead of cleanly short-circuiting or re-producing the prior artifact set. Sibling Issues #3172 (terminal-state pre-flight), #3174 (tag pre-flight), and #3175 (branch-tip zero-diff short-circuit) close the product behavior. Without an end-to-end harness scenario, future regressions of the apex-level idempotency contract would only be caught by dogfood — too late, too noisy. A scripted `FakeProvider` scenario gives us a deterministic CI-time net that fails fast on regression.

## Proposed Approach
Author one new scenario directory under [tests/harness/scenarios/](tests/harness/scenarios/) following the existing layout (cf. [tests/harness/scenarios/close_out_happy_path/scenario.yaml](tests/harness/scenarios/close_out_happy_path/scenario.yaml) and [tests/harness/scenarios/cascade_remedy_no_stale/scenario.yaml](tests/harness/scenarios/cascade_remedy_no_stale/scenario.yaml) as the closest structural references). Suggested slug: `apex_rerun_idempotency`.

The scenario:
1. Drives the apex workflow once end-to-end with a scripted `FakeProvider` transcript producing a normal happy-path artifact set (planned → actioned → implemented → closed-out, or whatever the configured apex pipeline lands at).
2. Snapshots the run-1 artifact set (branch tips, tags, generated docs / files under the scenario's working tree) into a fixture or in-memory baseline.
3. Re-dispatches the apex driver against the same root with a second `FakeProvider` transcript that expects EITHER (a) the zero-diff short-circuit terminal state introduced by #3175, OR (b) a re-execution that produces a byte-identical artifact set vs. the run-1 snapshot.
4. Asserts negatively: the run MUST NOT enter `scope_revise_cap_gate` (the AB#3127 regression signature). The assertion should name that state explicitly so a future regression's failure message points at the right ADR.
5. Wires into the existing scenario discovery / runner — no new harness plumbing; uses the .NET shim for `script:` nodes and the `polyphony-harness` skill conventions verbatim.

No product-code changes. No new harness plumbing. Single scenario directory, single `scenario.yaml`, plus any required transcript / fixture files.

## Dependencies
- **#3175** — Branch-tip zero-diff short-circuit at apex dispatch. Provides the "known-good terminal" branch this scenario asserts on. Hard dependency for the short-circuit assertion path.
- **#3172** — Apex-driver pre-flight refusal on terminal work-item state. Influences the second-run terminal.
- **#3174** — Apex-driver pre-flight tag check for previously-processed work items. Influences the second-run terminal.

This Issue is intentionally sequenced after the three above per the title ("deferred"). It is plannable now and indivisible, but execution should be gated until the upstream behavior lands.
