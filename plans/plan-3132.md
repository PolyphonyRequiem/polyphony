# Plan — `research:` profile config + storage abstraction (AB#3132)

## Summary
Add a new `research:` block to [.polyphony-config/profile.yaml](.polyphony-config/profile.yaml) (schema, deserialization, and validation) and a storage abstraction that can read/write/list documents in a sibling research repository. Auth defaults to **reusing the platform-router's existing credential resolution** (the existing `gh` / `az` token resolvers in [src/Polyphony/Infrastructure/Processes/GhTokenResolver.cs](src/Polyphony/Infrastructure/Processes/GhTokenResolver.cs) and [src/Polyphony/Infrastructure/AzureDevOps/AdoTokenResolver.cs](src/Polyphony/Infrastructure/AzureDevOps/AdoTokenResolver.cs)); the schema also accepts an optional explicit auth override for cross-platform cases without breaking the default path.

This is the foundation issue for Epic AB#3131 — every later child (the research sub-workflow, archivist curation, architect integration) depends on these two surfaces existing and round-tripping.

## Problem / Motivation
The research-augmented planning cycle (Epic AB#3131) needs:
1. A declarative way for a repo to opt into research without code changes — i.e. a profile entry.
2. A code-side abstraction that can durably write curated articles (and read them back) to a **sibling** repo, without inventing new secret plumbing for the common same-platform case.

No such config block or storage surface exists today. `FacetProfileConfig` ([src/Polyphony/Configuration/FacetProfileConfig.cs](src/Polyphony/Configuration/FacetProfileConfig.cs)) and `ConfigValidator` ([src/Polyphony/Configuration/ConfigValidator.cs](src/Polyphony/Configuration/ConfigValidator.cs)) are the model for how to add this cleanly. The `gh` and `az` token resolvers already used by polyphony's platform router are the natural credential source.

## Proposed Approach

### Surface 1 — `research:` profile schema + validation
- Introduce `ResearchProfileConfig` (YAML-friendly DTO, mutable string properties, `[YamlMember]` aliases) under [src/Polyphony/Configuration/](src/Polyphony/Configuration/), mirroring the `FacetProfileConfig` pattern.
- Fields (initial set, drawn from plan-3131 §Resolved Defaults):
  - `enabled: bool` (default false)
  - `remote:` block — `platform: github|azuredevops`, `owner`, `repo`, `branch` (default `main`), optional `path_prefix`
  - `auth:` optional block — when omitted, fall back to platform-router resolution; when present, accepts an explicit reference (e.g. `{ kind: env, var: GH_TOKEN }` or `{ kind: gh-cli }` / `{ kind: az-cli }`) so cross-platform cases (source-on-ADO + research-on-GH or vice versa) are expressible.
  - `escalation_cap: int` (default 1) — hooked here even though Issue 3 will read it, so the schema is stable from the start (avoids a schema churn in pg-3).
- Wire into top-level profile loader; add a canonical strongly-typed `ResearchProfile` record (immutable) and a `Get…` extension method, mirroring `ProcessConfigFacetExtensions`.
- Extend [ConfigValidator.cs](src/Polyphony/Configuration/ConfigValidator.cs) with a new `V-` rule: when `research.enabled = true`, `remote.platform/owner/repo` must be present; if `auth.kind` is supplied it must be a known kind. Additive only — does not change existing V-rules.

### Surface 2 — Storage abstraction
- Introduce `IResearchStorage` under `src/Polyphony/Infrastructure/Research/` with three operations: `Task<IReadOnlyList<ResearchEntry>> ListAsync(string pathPrefix, CancellationToken)`, `Task<ResearchDocument> ReadAsync(string path, CancellationToken)`, `Task WriteAsync(ResearchDocument doc, CancellationToken)`.
- Two implementations:
  - `GitHubResearchStorage` — uses the existing `IGhClient` ([src/Polyphony/Infrastructure/Processes/IGhClient.cs](src/Polyphony/Infrastructure/Processes/IGhClient.cs)) under the hood, so credentials come from `GhTokenResolver` automatically.
  - `AzureDevOpsResearchStorage` — uses the existing `IAdoClient` ([src/Polyphony/Infrastructure/AzureDevOps/IAdoClient.cs](src/Polyphony/Infrastructure/AzureDevOps/IAdoClient.cs)), credentials via `AdoTokenResolver`.
- A `ResearchStorageFactory` selects the implementation from `ResearchProfile.Remote.Platform`. When the optional explicit auth override is present, the factory plumbs an alternative resolver for that single client instance (does **not** mutate the global router) — this is the cross-platform escape hatch, defaulted off.
- Register both implementations + factory in [PolyphonyServiceRegistration.cs](src/Polyphony/Infrastructure/PolyphonyServiceRegistration.cs).
- Round-trip test: write a small markdown doc, list it, read it back; assert content equality. Run against a real sibling github repo in an integration-tier test (gated by env var so CI without creds skips it) plus unit tests with a fake `IGhClient` for the deterministic path.

### Sequencing within this Issue
Two Tasks, sequential: schema lands first (small, additive, validator-bounded), then storage builds on the typed `ResearchProfile` record.

## Acceptance Criteria
- [ ] `research:` block parses and validates from `.polyphony-config/profile.yaml`; missing/invalid blocks produce a clear V-rule diagnostic when `enabled=true`.
- [ ] `IResearchStorage` round-trips (read/write/list) against a sibling github repo using platform-router credentials, with no environment changes beyond what platform-router already requires.
- [ ] Schema exposes an optional explicit `auth` override but defaults to platform-router reuse — no new secret plumbing in the common path.
- [ ] Build passes with zero errors and warnings.
- [ ] All existing tests pass; new unit + integration tests cover schema parsing, validator behavior, and storage round-trip.
