# Wire `item_satisfied` as ADO transition trigger (close PR #6 carry-forward)

## Strategic Objective

Close the carry-forward gap left after PRs #6 (#220) and #9 (#221): make `polyphony validate --event item_satisfied --work-item N` return `IsValid=true` for every terminal-eligible work item type, so that the terminal-satisfied corridor in `apex-item-dispatch.yaml` actually advances the ADO work item state instead of silently skipping. Today the per-item state machine declares the apex complete while ADO still shows the prior state — this Epic restores that observable closure.

## Success Criteria

- `polyphony validate --event item_satisfied --work-item N` returns `IsValid=true` for every terminal-eligible work item type configured in [.conductor/process-config.yaml](.conductor/process-config.yaml).
- [.conductor/process-config.yaml](.conductor/process-config.yaml) contains an `item_satisfied` event mapping under `transitions:` for the appropriate active→terminal state pair on each terminal-eligible type.
- At least one unit test under [tests/Polyphony.Tests/](tests/Polyphony.Tests/) exercises the new mapping and asserts `IsValid=true` for `--event item_satisfied` on at least one type.
- `lint-conductor-validate` (the existing lint script) passes against the updated config.
- Dogfood: a real terminal-satisfied corridor execution (apex-item-dispatch.yaml) is observed to advance the ADO work item state, not silently skip.

## Scope

### In Scope

- Adding the `item_satisfied` event under `transitions:` for each terminal-eligible type in [.conductor/process-config.yaml](.conductor/process-config.yaml).
- Adding/extending a unit test in [tests/Polyphony.Tests/](tests/Polyphony.Tests/) that proves the validator now accepts the event.
- Re-running `lint-conductor-validate` and confirming green.
- A dogfood run that demonstrates the ADO state actually advances when the corridor fires.

### Out of Scope

- Behavior changes beyond enabling the event mapping (validator semantics, router logic, driver logic — all already done in PRs #6 and #9).
- Changes to `lifecycle-router.ps1`.
- Changes to apex-driver outer-loop logic.
- Wiring corresponding events for non-terminal kinds (e.g. `item_planned`, `item_implemented`) — separate concern, separate work item.

## Child Issues

- task-1 — Wire `item_satisfied` event mapping + unit test + lint — implements the config edit and test that satisfy criteria 1–4.
- task-2 — Dogfood verification of terminal-satisfied ADO state advance — satisfies criterion 5 and serves as the closing evidence for the Epic.

## Risks

- **Type coverage drift.** The work item description names `Epic, User Story, Task, Bug` as terminal-eligible, but the visible [.conductor/process-config.yaml](.conductor/process-config.yaml) declares `Epic, Issue, Task`. The implementing Issue must enumerate exactly the terminal-eligible types from the live config, not from a hard-coded list — see open question below.
- **State-name assumptions.** Per Polyphony first principle P5 (no hard-coded state names), the config edit must use the literal state names already declared per-type rather than assume `In Progress → Done` everywhere.
