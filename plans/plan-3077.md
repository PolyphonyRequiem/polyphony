# Wiki polish: Johnny-Decimal, Obsidian compat, citation rigor, `INDEX.md`

## Summary
Add the Obsidian-friendly polish layer the archivist (#3075) applies when promoting kept articles from `research/scratch/<apex-id>/` into the sibling research repo: Johnny-Decimal path layout (informed by collected reference material), YAML frontmatter + wiki-link compatibility, mandatory citation fields (URL + capture date + freshness) enforced by a standalone citation linter, and an auto-maintained `INDEX.md`. Outcome: kept articles are deterministically discoverable in Obsidian and verifiable by a citation check that needs no polyphony install.

## Problem / Motivation
Parent epic #3071 promises that kept articles "graduate to a sibling research repo (Obsidian-friendly markdown, Johnny-Decimal layout, academic-style citations with direct links and freshness timestamps)." #3075 builds the archivist that performs the write, but without an enforced layout + frontmatter + citation contract the wiki degrades into ad-hoc files that future research-assistant passes cannot reliably search, link, or trust. We need the polish layer locked in *before* the archivist starts populating real archives at scale, otherwise we pay a migration cost later.

## Proposed Approach
Five cohesive pieces, all landing in one PG. Concrete file paths inside the sibling repo are defined here; paths inside the polyphony repo (where the archivist code lives) are deferred until #3075's scaffolding lands.

1. **Johnny-Decimal research + reference pack.** Before locking the layout, do a focused pass on the canonical Johnny-Decimal sources (johnnydecimal.com, the small-business guide, common pitfalls around area sprawl and category renumbering) and capture the relevant excerpts as reference markdown files committed under `wiki/.reference/johnny-decimal/` in the sibling repo. The archivist agent loads these as guidance when it has to make a path decision ("is this a new category or a sub-topic of an existing one?"). Reference files carry their own frontmatter and source URLs so they pass the citation check.

2. **Johnny-Decimal layout convention — recommended initial split.** The user asked for more on the initial split. The parent epic's keep-decision axes are domain, codebase, technology stacks, ecosystem (4 axes). Two viable shapes:
   - **A. 1:1 axes-to-areas (recommended).** `10-19 Domain / 20-29 Codebase / 30-39 Tech Stacks / 40-49 Ecosystem`. Pros: archivist routing is trivial (the keep-decision already names the axis, so the area falls out for free); only 4 areas means the registry stays scannable; Johnny-Decimal gives us 10 categories per area (e.g. `21 Workflow Engine`, `22 Storage`, `23 CLI Surface` under Codebase) which is where the real bucketing work happens. Cons: "Domain" is broad and may need its categories curated early to avoid a dumping ground.
   - **B. Finer-grained seed (e.g. 6–8 areas splitting Codebase into Engine / CLI / Tooling).** Pros: shorter category lists per area. Cons: archivist needs a second routing decision (axis → which sub-area?); area registry churns more in the first months; renumbering an area later is the most expensive J-D mistake.
   - **Recommendation: ship A.** Categories under each area absorb the real growth, and the registry stays mechanically aligned with the keep-decision. If "Domain" gets crowded (>8 categories) we can split it later with a documented migration — that pain is bounded because area renumbering touches paths but not wiki-links (links are basename-based; see piece 3).
   - The convention is codified as the archivist's path-resolution rule, with a `wiki/.areas.yaml` registry pinning area+category numbers so they stay stable across runs. **Registry policy: the archivist fails closed on unknown area or category** — it does not auto-append. An unresolvable keep-decision surfaces back through the archivist's renegotiation path so a human (or the parent planning loop) authors the registry entry deliberately. Auto-append was rejected because a silently-grown registry is the exact "area sprawl / category drift" failure mode the J-D reference pack warns against.

3. **Obsidian compatibility.** Every kept article gets YAML frontmatter (`title`, `tags`, `source_url`, `captured`, `freshness`, `area`, `category`, `id`). Internal references use `[[wiki-link]]` syntax against article basename. Because the Johnny-Decimal id is part of the basename (`11.03 conductor-routing.md` → `[[11.03 conductor-routing]]`), links survive area renumbering as long as the id stays stable; the archivist treats id reassignment as a deliberate migration with link-rewrite, not a routine write. The archivist normalizes inbound markdown links to wiki-links during the keep step. **`captured` vs `freshness`:** `captured` is the immutable date the source content was first pulled into the wiki (set once on initial archive, never rewritten); `freshness` is the date the archivist (or a human) last re-verified the source against the live URL and confirmed it still reflects current truth — bumped on every re-verification, set equal to `captured` on initial archive, and set to the literal `evergreen` for sources whose content does not meaningfully decay (e.g. canonical specs, historical post-mortems). **Staleness behaviour is deferred to a follow-up:** this PG enforces field shape only; no threshold check, no auto-refresh, no archivist warning on old freshness. A future issue under #3071 will pick a threshold and decide whether the archivist re-fetches or just flags.

4. **Citation rigor — standalone script (Python).** A small standalone Python script (single file, stdlib-only, no polyphony dependency) ships in the sibling repo at `wiki/.tools/check-citations` and is invoked both by the archivist (shelled out during the commit step) and by the sibling repo's CI. **Python pinned** because the existing polyphony tooling footprint already assumes Python is available on contributor machines and CI runners; standardising avoids a Node toolchain just for this script. **Frontmatter parsing under stdlib-only:** the script ships its own minimal hand-rolled frontmatter parser — it reads the leading `---`-delimited block at the top of the file and parses only the flat `key: value` shape we actually emit (no nested mappings, no multi-line scalars, no anchors). Tags are parsed as a single-line flow sequence (`tags: [a, b, c]`). The frontmatter we control is mechanically generated by the archivist, so we can keep the schema narrow enough to dodge full YAML; if a future field needs richer YAML we relax the constraint to allow PyYAML rather than grow the hand-rolled parser. It verifies every kept article's frontmatter contains `source_url` (non-empty absolute URL), `captured: YYYY-MM-DD`, and `freshness: YYYY-MM-DD` or the literal `evergreen`. **The script also resolves wiki-links:** every `[[wiki-link]]` in a kept article must match the basename (sans `.md`) of an existing article in the tree; unresolved links exit non-zero. This makes Obsidian-vault correctness a binary CI check rather than a manual "open it and look" gate. Standalone keeps the sibling repo's CI free of a polyphony install; the archivist still gets inline enforcement by shelling out to the same script.

5. **`INDEX.md` maintenance.** Regenerated deterministically from the on-disk tree + frontmatter on every archivist write — grouped by area and category, sorted by Johnny-Decimal id, with title pulled from frontmatter. No hand edits; the file carries a generator banner. Inline-only (no separate hook/job) keeps the contract atomic — if a write succeeds, INDEX is correct; the only drift vector is hand-edits to the wiki, which the archivist heals on its next write.

All pieces consume the storage abstraction from #3072 and slot into the archivist commit path from #3075 — no new platform plumbing.

## Acceptance Criteria
- [ ] Johnny-Decimal reference pack lives under `wiki/.reference/johnny-decimal/` with at least the canonical site material and a short "common pitfalls" digest, each file passing the citation check.
- [ ] Initial area registry seeds the four-axis split (Domain / Codebase / Tech Stacks / Ecosystem) at `wiki/.areas.yaml`; archivist refuses to write a kept article without a resolvable area/category/id and surfaces unknown area/category back through renegotiation rather than auto-appending.
- [ ] Every kept article has YAML frontmatter with the required fields; internal cross-references use `[[wiki-link]]` syntax; the citation script verifies every `[[wiki-link]]` in a kept article resolves to an existing article basename in the tree (exit non-zero on any unresolved link).
- [ ] Standalone citation-check script is a single Python file (stdlib-only, no polyphony dependency) in the sibling repo, exits non-zero on any violation, and is invoked both inline by the archivist and by the sibling repo's CI.
- [ ] `INDEX.md` is regenerated deterministically on every archivist write; the file is byte-identical when regenerated against an unchanged tree (path-level idempotency) AND byte-identical when regenerated after a frontmatter-only change that does not affect any title, area, category, or id (frontmatter-drift idempotency).
- [ ] Build passes with zero errors and warnings.
- [ ] All existing tests pass; new tests cover citation-check pass/fail cases (including unresolved wiki-link), INDEX regeneration determinism (both idempotency variants above), Johnny-Decimal path resolution, and a malformed `tags:` flow-sequence input to the citation script's frontmatter parser (must fail loudly with a non-zero exit and a diagnostic, not silently fall back to treating the value as a string).

## Resolved design defaults (from prior round)
- **Citation check execution surface:** standalone script in the sibling repo, shelled out by the archivist; sibling CI does not need polyphony installed.
- **INDEX.md regeneration:** inline during archivist write only; no separate hook/job. Hand-edits drift until the next archivist run, which is acceptable because INDEX is generator-banner-marked.
- **Wiki-link basename stability:** basename-based links with id embedded; id reassignment is a deliberate migration with link rewrite, not a routine write. Acceptable for first pass; redirect/alias mechanism deferred until we observe real reassignment pain.
- **Id-reassignment link-rewrite mechanism:** explicitly deferred to a follow-up issue under #3071 (mirroring the `freshness` threshold deferral). The first id reassignment will require either a one-off rewrite tool or a documented manual procedure; this PG does not pick which, only commits that the contract here (basename-with-id) keeps the future option open.
- **Registry growth policy (this round):** archivist fails closed on unknown area/category and surfaces renegotiation; no auto-append.
- **Citation script language (this round):** Python, single file, stdlib-only; ships its own minimal hand-rolled frontmatter parser scoped to the flat schema the archivist emits, with the option to relax to PyYAML if future fields need richer YAML.
- **`freshness` staleness threshold (this round):** explicitly deferred to a follow-up issue under #3071; this PG enforces shape only.
