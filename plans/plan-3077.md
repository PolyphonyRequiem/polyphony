# Wiki polish: Johnny-Decimal, Obsidian compat, citation rigor, `INDEX.md`

## Summary
Add the Obsidian-friendly polish layer the archivist (#3075) applies when promoting kept articles from `research/scratch/<apex-id>/` into the sibling research repo: Johnny-Decimal path layout (informed by collected reference material), YAML frontmatter + wiki-link compatibility, mandatory citation fields (URL + capture date + freshness) enforced by a standalone citation linter, and an auto-maintained `INDEX.md`. Outcome: kept articles are deterministically discoverable in Obsidian and verifiable by a citation check that needs no polyphony install.

## Problem / Motivation
Parent epic #3071 promises that kept articles "graduate to a sibling research repo (Obsidian-friendly markdown, Johnny-Decimal layout, academic-style citations with direct links and freshness timestamps)." #3075 builds the archivist that performs the write, but without an enforced layout + frontmatter + citation contract the wiki degrades into ad-hoc files that future research-assistant passes cannot reliably search, link, or trust. We need the polish layer locked in *before* the archivist starts populating real archives at scale, otherwise we pay a migration cost later.

## Proposed Approach
Five cohesive pieces, all landing in one PG. Concrete file paths inside the sibling repo are defined here; paths inside the polyphony repo (where the archivist code lives) are deferred until #3075's scaffolding lands.

1. **Johnny-Decimal research + reference pack.** Before locking the layout, do a focused pass on the canonical Johnny-Decimal sources (johnnydecimal.com, the small-business guide, common pitfalls around area sprawl and category renumbering) and capture the relevant excerpts as reference markdown files committed under `wiki/.reference/johnny-decimal/` in the sibling repo. The archivist agent loads these as guidance when it has to make a path decision ("is this a new category or a sub-topic of an existing one?"). Reference files carry their own frontmatter and source URLs so they pass the citation check.

2. **Johnny-Decimal layout convention — recommended initial split.** The user asked for more on the initial split. The parent epic's keep-decision axes are domain, codebase, technology stacks, ecosystem (4 axes). Two viable shapes:
   - **A. 1:1 axes-to-areas (recommended).** `10-19 Domain / 20-29 Codebase / 30-39 Tech Stacks / 40-49 Ecosystem`. Pros: archivist routing is trivial (the keep-decision already names the axis, so the area falls out for free); only 4 areas means the registry stays scannable; Johnny-Decimal gives us 10 categories per area (e.g. `21 Workflow Engine`, `22 Storage`, `23 CLI Surface` under Codebase) which is where the real bucketing work happens. Cons: "Domain" is broad and may need its categories curated early to avoid a dumping ground.
   - **B. Finer-grained seed (e.g. 6–8 areas splitting Codebase into Engine / CLI / Tooling).** Pros: shorter category lists per area. Cons: archivist needs a second routing decision (axis → which sub-area?); area registry churns more in the first months; renumbering an area later is the most expensive J-D mistake.
   - **Recommendation: ship A.** Categories under each area absorb the real growth, and the registry stays mechanically aligned with the keep-decision. If "Domain" gets crowded (>8 categories) we can split it later with a documented migration — that pain is bounded because area renumbering touches paths but not wiki-links (links are basename-based; see piece 3).
   - The convention is codified as the archivist's path-resolution rule, with a `wiki/.areas.yaml` registry pinning area+category numbers so they stay stable across runs. New categories require a registry entry.

3. **Obsidian compatibility.** Every kept article gets YAML frontmatter (`title`, `tags`, `source_url`, `captured`, `freshness`, `area`, `category`, `id`). Internal references use `[[wiki-link]]` syntax against article basename. Because the Johnny-Decimal id is part of the basename (`11.03 conductor-routing.md` → `[[11.03 conductor-routing]]`), links survive area renumbering as long as the id stays stable; the archivist treats id reassignment as a deliberate migration with link-rewrite, not a routine write. The archivist normalizes inbound markdown links to wiki-links during the keep step.

4. **Citation rigor — standalone script.** A small standalone script (Python or Node, single file, no polyphony dependency) ships in the sibling repo at `wiki/.tools/check-citations` and is invoked both by the archivist (shelled out during the commit step) and by the sibling repo's CI. It verifies every kept article's frontmatter contains `source_url` (non-empty absolute URL), `captured: YYYY-MM-DD`, and `freshness: YYYY-MM-DD` or the literal `evergreen`. Standalone keeps the sibling repo's CI free of a polyphony install; the archivist still gets inline enforcement by shelling out to the same script.

5. **`INDEX.md` maintenance.** Regenerated deterministically from the on-disk tree + frontmatter on every archivist write — grouped by area and category, sorted by Johnny-Decimal id, with title pulled from frontmatter. No hand edits; the file carries a generator banner. Inline-only (no separate hook/job) keeps the contract atomic — if a write succeeds, INDEX is correct; the only drift vector is hand-edits to the wiki, which the archivist heals on its next write.

All pieces consume the storage abstraction from #3072 and slot into the archivist commit path from #3075 — no new platform plumbing.

## Acceptance Criteria
- [ ] Johnny-Decimal reference pack lives under `wiki/.reference/johnny-decimal/` with at least the canonical site material and a short "common pitfalls" digest, each file passing the citation check.
- [ ] Initial area registry seeds the four-axis split (Domain / Codebase / Tech Stacks / Ecosystem) at `wiki/.areas.yaml`; archivist refuses to write a kept article without a resolvable area/category/id.
- [ ] Every kept article has YAML frontmatter with the required fields; internal cross-references use `[[wiki-link]]` syntax; opening the sibling repo as an Obsidian vault resolves all internal links.
- [ ] Standalone citation-check script lives in the sibling repo, runs with no polyphony dependency, exits non-zero on any violation, and is invoked both inline by the archivist and by the sibling repo's CI.
- [ ] `INDEX.md` is regenerated deterministically on every archivist write; the file is byte-identical when regenerated against an unchanged tree (idempotency test).
- [ ] Build passes with zero errors and warnings.
- [ ] All existing tests pass; new tests cover citation-check pass/fail cases, INDEX regeneration determinism, and Johnny-Decimal path resolution.

## Resolved design defaults (from prior round)
- **Citation check execution surface:** standalone script in the sibling repo, shelled out by the archivist; sibling CI does not need polyphony installed.
- **INDEX.md regeneration:** inline during archivist write only; no separate hook/job. Hand-edits drift until the next archivist run, which is acceptable because INDEX is generator-banner-marked.
- **Wiki-link basename stability:** basename-based links with id embedded; id reassignment is a deliberate migration with link rewrite, not a routine write. Acceptable for first pass; redirect/alias mechanism deferred until we observe real reassignment pain.
