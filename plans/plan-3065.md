# Audit architect plan-level prompt for prose-vs-structured children alignment

## Strategic Objective

Close the ergonomic gap left by closed-loop §3.4(a) (PR #214, apex_facets marker) and PR #225 (F3, strict seed-children): the architect agent itself can still emit a plan whose markdown body declares child items in prose (under `## Child Issues` or narrative paragraphs) while leaving the structured `children` array empty. Today this is caught at seed-children time with a generic "ambiguous" error — one verb too late, no hint about which prose declarations leaked. Audit [.conductor/registry/prompts/architect-plan-level.md](.conductor/registry/prompts/architect-plan-level.md) against the §3.4(a) contract and either (a) strengthen specific sentences surgically, (b) ship a deterministic post-architect check verb (`polyphony plan check-children`), or (c) document that the existing language is sufficient.

## Success Criteria

- Audit findings recorded against [.conductor/registry/prompts/architect-plan-level.md](.conductor/registry/prompts/architect-plan-level.md) lines 279–291 and 358–363 vs. the §3.4(a) contract.
- A decision recorded on the deterministic check verb: ship it, or written rationale (overlap with F3/F4, latency, false-positive risk on prose like the audit narrative itself) for not.
- If prompt language changes: surgical edits only; any e2e architect tests under `tests/` that snapshot prompt content updated.
- If a new verb ships: tests under [tests/Polyphony.Tests/Commands/](tests/Polyphony.Tests/Commands/), registered in `PolyphonyJsonContext`, snake_case JSON output, exit codes per [src/Polyphony/ExitCodes.cs](src/Polyphony/ExitCodes.cs), `JsonOutputContractTests` updated.
- All work lands in a single PR.

## Scope

### In Scope
- Reading the current architect-plan-level prompt and comparing the children-contract language against §3.4(a).
- Surgical prompt edits if the audit finds ambiguity.
- Optional: a `polyphony plan check-children` verb that parses plan markdown for child-declaring headings/lists and cross-references the structured `children` array, with tests.
- Updating any prompt-snapshot tests if prompt text changes.

### Out of Scope
- F4 (PR-time lint scanning plan PR diffs) — separate polish item.
- Architect-prompt structural rewrites unrelated to the children contract.
- Changes to `seed-children` itself (already strict per PR #225).
- Other prompts in the registry beyond architect-plan-level.

## Indivisibility rationale

The work item explicitly notes: "This may be indivisible (one prompt edit, ship it) — declare apex_facets: [implementable] if so. The audit-decision-implementation arc fits in a single PR." The audit, decision, and (at most) one surgical prompt edit plus optionally one new CLI verb with tests cleanly fits in a single PR group (≤ ~2000 LoC, ≤ 50 files). Decomposing into Issues would add ceremony without informational value — the audit cannot be meaningfully scoped before it is performed, and the implementation surface is bounded by the acceptance criteria.

Declaring `apex_facets: [implementable]` per closed-loop §3.4(a) so the seeder treats the empty `children` array as deliberate.

## Risks

- A `check-children` verb may produce false positives when the plan markdown legitimately discusses child items in narrative form (e.g. this very audit, which references `## Child Issues` as an example of the failure mode). Mitigation: scope detection to specific structural patterns (headings + bulleted lists with link-or-id syntax) and document the heuristic in the verb's help text.
- Prompt edits risk drift from snapshot tests — mitigation already covered by acceptance criteria.
