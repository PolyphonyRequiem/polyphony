# Plan — Architect integration: `research_needs` emission + loop-back routing bypassing review (AB#3136)

## Summary
Wire the existing `architect` agent in [.conductor/registry/workflows/plan-level.yaml](.conductor/registry/workflows/plan-level.yaml) to (a) emit a structured `research_needs` payload, (b) trigger the research sub-workflow built by sibling issues 2–4, and (c) re-enter itself with the resulting findings without traversing the plan-review gate. Concludes with a harness/dogfood scenario that proves the architect → research → architect loop on a non-trivial prompt.

## Problem / Motivation
Today the architect has no escape valve for genuine knowledge gaps: it must either guess, raise an open question (which gates on a human), or push the gap downstream. Once the research sub-workflow lands (parent epic AB#3131, sibling issue-2 onwards), architects need a plumbed-in way to *use* it. The loop must bypass the plan-review gate — research findings are *context*, not commitment, so forcing each architect→research bounce through a human/PR review would make the cycle unusable.

This issue is the integration seam. It is also the issue whose acceptance criterion includes the **first unattended end-to-end dogfood proof** that the loop closes — i.e. it is where the parent epic earns its outcome.

## Proposed Approach

Three disjoint surfaces, implementable as three Tasks:

1. **Architect agent contract** — extend the `architect` step in [.conductor/registry/workflows/plan-level.yaml](.conductor/registry/workflows/plan-level.yaml) (≈ lines 531–597) with a `research_needs` array in its declared `output:` schema, matching the payload shape defined by sibling issue-2. Update [.conductor/registry/prompts/architect-plan-level.md](.conductor/registry/prompts/architect-plan-level.md) to (a) teach the architect when to emit `research_needs` vs. an open question vs. proceeding, and (b) add a new re-entry branch (alongside the existing `revise_counter`, `extract_parent_patch`, `open_questions_gate` branches) that fires when the prior step in `context.history` is the research sub-workflow, surfacing the findings document(s) to the architect for incorporation.

2. **Workflow wiring (loop-back, bypass review)** — change the architect's `routes:` so that when `architect.output.research_needs | length > 0`, it routes to a new `research_dispatch` node that invokes the research sub-workflow built by issue-2/3/4. After the sub-workflow returns, route directly back to `architect` — *never* to `open_questions_policy`, `ensure_plan_branch`, `write_plan`, or `plan_reviewer`. Wrap the loop in a counter (mirroring the `open_questions_counter` / `revise_counter` pattern) so a runaway architect cannot bounce forever; cap defaults are profile-overridable. Gate the entire path on the `research:` profile being configured — repos without a research config see no behavioural change.

3. **End-to-end proof scenario** — add either a path-coverage harness scenario under [tests/harness/scenarios/](tests/harness/scenarios/) (FakeProvider scripted to emit `research_needs` once, then proceed) **or** a dogfood-recipe document that drives a live architect → research → architect cycle on a non-trivial prompt and captures the run trace, depending on the answer to open question Q3 below. The acceptance criterion requires *one* of these; both are not in scope.

Dependency note: tasks 1+2 require the `research_needs` schema and the research sub-workflow entry-point name to be locked. Per the parent plan, sibling issue-2 owns the schema definition; this issue *consumes it unchanged*. If issue-2 has not landed in the consuming branch when implementation starts, either the implementer rebases on it or stubs the dispatch target with a clearly-labelled placeholder that fails closed.

## Acceptance Criteria
- [ ] An architect can emit `research_needs` (validated by the agent's `output:` schema) and the workflow runs the research sub-workflow when the `research:` profile block is present.
- [ ] The next architect iteration receives the findings without traversing `open_questions_gate`, `ensure_plan_branch`, `write_plan`, `open_plan_pr`, or `plan_reviewer`.
- [ ] A research-loop counter caps repeated `research_needs` emissions and surfaces a human gate (or proceeds) when the cap is hit, mirroring `open_questions_counter` / `revise_counter`.
- [ ] Repos with no `research:` profile block see no behavioural change in `plan-level.yaml` execution.
- [ ] A harness scenario or dogfood scenario demonstrates the full architect → research → architect loop on a non-trivial prompt without human intervention.
- [ ] Build passes with zero errors and warnings.
- [ ] All existing tests pass; new tests cover the new routing edges and the re-entry prompt branch.
