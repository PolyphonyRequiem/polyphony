# Research sub-workflow & `research-assistant` agent

## Summary

Stand up the first-pass research capability: a new conductor sub-workflow registered under [.conductor/registry/workflows/](.conductor/registry/workflows/) and a lightweight `research-assistant` agent that reads the kept-archive (via the storage abstraction from #3108) and emits a structured findings document with citations into per-feature scratch on the feature branch. This is the foundational rung of the three-tier researcher pattern locked in parent Epic #3107 — sufficiency judgement, escalation, curation, and architect-side loop-back are all explicitly out of scope and tracked under sibling Issues #3110/#3111/#3112.

## Problem / Motivation

Per the parent Epic, polyphony's planning cycle is a closed loop today — architects can only think with what's in their prompt. The Epic's locked design is a three-tier researcher pattern with archive-only first pass, sufficiency judging, and deep-research escalation. Before any of that machinery can come online, we need the bottom rung in place: a callable sub-workflow with the lightweight assistant that produces consumable findings. Without this Issue, no other Issue in the Epic has anything to plug into.

## Proposed Approach

**Sub-workflow file.** Add a new YAML at `.conductor/registry/workflows/research.yaml` following the conventions of existing sub-workflows in that directory (e.g. [actionable.yaml](.conductor/registry/workflows/actionable.yaml), [plan-level.yaml](.conductor/registry/workflows/plan-level.yaml)). It accepts a research request (topic / question / parent work item id / scratch slug) from a parent architect workflow and returns a path to the findings document. Register it in [.conductor/registry/index.yaml](.conductor/registry/index.yaml) so any architect can invoke it.

**Agent prompt.** Add `research-assistant` agent prompt under `.conductor/registry/prompts/` (sibling to [architect-plan-level.md](.conductor/registry/prompts/architect-plan-level.md)). The prompt is scoped to: (a) call the kept-archive read API exposed by the storage abstraction from #3108, (b) score relevance, (c) emit a findings markdown document with citations (source URL + capture date — minimum bar). Use a lightweight model (the workflow YAML pins the cheap tier).

**Scratch storage.** Findings are written under `research/scratch/<slug>/findings.md` on the current feature branch. Slug is derived from the requesting work item / run id (exact format below — see open question). No pruning logic in this Issue; pruning belongs to architect-integration / close-out.

**Findings format.** A markdown document with a fixed header (topic, requested-by work item, capture timestamp), then a list of cited articles each with: archive path, source URL, capture date, one-paragraph relevance summary. Downstream architects (#3112) consume this format.

**Test.** End-to-end exercise with a mocked archive (a fixture directory pretending to be the kept-archive) — workflow runs, agent reads fixture, findings.md is produced and validated against the format spec.

## Acceptance Criteria

- [ ] A research sub-workflow is registered in [.conductor/registry/index.yaml](.conductor/registry/index.yaml) and invocable from a parent architect workflow.
- [ ] `research-assistant` agent prompt exists under `.conductor/registry/prompts/` and is bound to the lightweight model tier in the workflow YAML.
- [ ] Agent reads the kept-archive via the abstraction introduced by #3108 (no direct filesystem coupling) and emits a structured findings document.
- [ ] Findings written to `research/scratch/<slug>/findings.md` on the feature branch with a fixed, documented header + per-article schema.
- [ ] Each cited article includes source URL and capture date (minimum bar).
- [ ] End-to-end test exercises the workflow against a mocked kept-archive fixture and asserts findings.md matches the format spec.
- [ ] Workflow YAML passes the existing lint suite under [.conductor/registry/tests/](.conductor/registry/tests/) (add a new `lint-research.Tests.ps1` modelled on siblings).
- [ ] Build passes with zero errors and warnings.
- [ ] All existing tests pass; new tests cover the new workflow + agent.
