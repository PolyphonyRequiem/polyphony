# `archivist` curation pass — keep / discard / expand

## Summary

Add the archivist curation step that runs at the end of every research sub-workflow invocation. The archivist is an LLM agent that judges each scratch article for keep / discard / expand against domain, codebase, tech stacks, ecosystem, and linkability; kept articles are graduated into the configured sibling research repo via the storage abstraction landed in Issue #3108. Scope is bounded enough to land as a single PG behind its dependencies.

## Problem / Motivation

The research sub-workflow (Issue #3109) produces scratch articles on the feature branch. Without a curation step those articles either rot in scratch or get force-pushed wholesale into the wiki — both fail the design intent ("kept articles graduate; scratch is ephemeral"). The archivist closes that loop: every research pass ends with explicit, recorded decisions and a clean hand-off to the sibling-repo writer.

This is the third of six children under Epic #3107 (research-augmented planning cycle). It depends on:

- **#3108** — research storage abstraction & `research:` profile block (gives us the sibling-repo write seam and platform-router integration).
- **#3109** — research sub-workflow & `research-assistant` agent (gives us the scratch directory contract and the workflow node where the archivist plugs in).

## Proposed Approach

Four tightly-coupled pieces, all landing in one PG once the dependencies are in `main`:

1. **`archivist` agent prompt + workflow node.** New agent prompt under `.polyphony-config/agents/` (path mirrors the existing agent layout); wired as the terminal node of the research sub-workflow YAML defined in Issue #3109. Input: the scratch directory listing + each article's frontmatter and body. Output: a structured decision record per article (`keep` | `discard` | `expand`) with rationale, plus optional expansion guidance for the `expand` path.
2. **Decision recording.** The archivist's structured output is persisted as a sidecar JSON (one entry per scratch article) so downstream nodes — and tests — can assert on decisions deterministically. Format follows the conductor structured-output convention already used by other agent nodes.
3. **Sibling-repo writer.** A deterministic step (PowerShell helper under `scripts/`, consistent with `polyphony-workflow-author` guidance) that consumes the decision record and, for every `keep`, calls the storage abstraction from #3108 to write the article into the sibling research repo as Obsidian-friendly markdown. `discard` is a no-op beyond the recorded decision; `expand` records guidance for the next research pass but does NOT itself re-trigger the researcher (escalation flow is owned by Issue #3110).
4. **Scratch hand-off contract.** Document — in the research sub-workflow YAML and in the agent prompt — that scratch lifecycle is: created in #3109, curated here, pruned at apex close. Pruning itself is owned by Issue #3112 (architect-side wiring) per the parent Epic's decomposition; this Issue stops at recording the contract, not implementing the prune.

Tests cover all three decision branches plus the sibling-repo write path. Where the storage abstraction supports a fake/in-memory backend (delivered by #3108), tests use it; otherwise we use the harness under [tests/harness/](tests/harness/) with a scripted `FakeProvider` for the agent boundary, per `polyphony-harness` guidance.
