# `researcher` sufficiency check & `deep-researcher` escalation

## Summary

Layer the cost-tiered escalation tier onto the research sub-workflow established by sibling Issue #3109. Add two agents — `researcher` (judges sufficiency of the assistant's first pass and decides whether to escalate) and `deep-researcher` (extended-toolset researcher invoked only on escalation) — wired into the research sub-workflow with a bounded escalation loop so the pipeline cannot spin indefinitely.

## Problem / Motivation

The parent Epic (#3107) locks in a three-tier researcher pattern as a cost-tiering strategy: cheap archive-only first pass (`research-assistant`, delivered by #3109), a sufficiency judgment, and an expensive extended-tool pass only when the cheap pass came up short. Without the middle judgment and the escalation tier, every research need either gets the cheap answer or an unbounded recursion — neither acceptable. This Issue closes that gap so the research subsystem can actually answer non-trivial `research_needs` without burning budget on every call.

## Proposed Approach

**Where this lands.** The research sub-workflow YAML and `research-assistant` agent are produced by sibling Issue #3109. This Issue extends that sub-workflow in place — it does not create a new top-level workflow.

**New agents** (under [workflows/agents/](workflows/agents/) following the polyphony agent-prompt convention used by `architect`, `reviewer`, etc.):

- `researcher.agent.md` — receives the original `research_needs` plus the `research-assistant`'s findings. Emits a structured decision: `{ decision: "sufficient" | "escalate", rationale: string, refined_needs?: string }`. When `sufficient`, the sub-workflow exits with the assistant's findings as the answer. When `escalate`, the sub-workflow advances to the deep tier with `refined_needs` as the brief.
- `deep-researcher.agent.md` — same input contract as `research-assistant` plus the `refined_needs` and the assistant's prior findings (so it doesn't redo the cheap work). Has access to the extended toolset (web search, GitHub search, MCP-provided integrations) — the actual tool wiring is per-agent config the polyphony runtime already supports.

**Sub-workflow wiring** (in the research sub-workflow YAML produced by #3109):

1. After the `research-assistant` node, route to a `researcher` node.
2. From `researcher`, branch on `decision`:
   - `sufficient` → exit sub-workflow with the assistant's findings.
   - `escalate` → invoke `deep-researcher` with the refined needs.
3. After `deep-researcher` runs, exit the sub-workflow with the deep findings (no second sufficiency judgment in this Issue — single escalation is the bound; see below).

**Bounded escalation policy.** A single escalation hop is the bound (assistant → judge → deep-researcher → done). No loop back to the judge. This is the simplest bound that preserves the cost-tiered shape and is trivially terminating; any future tightening (e.g., a max-N retry across the deep tier) can be added without re-architecting. The bound lives in the sub-workflow YAML structure itself rather than as a runtime knob — documented in the sub-workflow header comment and called out in the open questions below in case the user wants it config-driven instead.

**Tests.** Extend the sub-workflow scenarios under [tests/harness/scenarios/](tests/harness/scenarios/) (the path-coverage harness — see the `polyphony-harness` skill) with two new scenarios:

- `research-sufficiency-no-escalate.yaml` — `researcher` returns `sufficient`; assert `deep-researcher` is never invoked and the assistant findings are returned.
- `research-sufficiency-escalate.yaml` — `researcher` returns `escalate`; assert `deep-researcher` runs exactly once with the refined needs and its findings are returned.

The `FakeProvider` script in each scenario stubs the agent responses; no real LLM calls.

## Acceptance Criteria

- [ ] `researcher` agent exists at `workflows/agents/researcher.agent.md` and produces an explicit `sufficient` / `escalate` decision with a recorded rationale on every pass.
- [ ] `deep-researcher` agent exists at `workflows/agents/deep-researcher.agent.md`, runs only on the escalate branch, and is configured with the extended toolset (web search, GitHub search, MCP-provided integrations).
- [ ] The research sub-workflow YAML (delivered by #3109) is extended with the judge node and the conditional escalate edge to `deep-researcher`.
- [ ] Escalation is bounded: a single escalation hop, no loop back. The bound is documented in the sub-workflow header comment.
- [ ] Harness scenario `research-sufficiency-no-escalate` passes and asserts `deep-researcher` is not invoked.
- [ ] Harness scenario `research-sufficiency-escalate` passes and asserts `deep-researcher` is invoked exactly once with the refined needs.
- [ ] Build passes with zero errors and warnings.
- [ ] All existing tests pass; new tests cover both no-escalation and escalation paths.
