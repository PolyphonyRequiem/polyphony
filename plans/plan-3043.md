# Epic Plan — Document `Invoke-PolyphonySdlc.ps1` and worktree convention

## Strategic Objective

Close the discoverability gap between PR #155 (the canonical [`scripts/Invoke-PolyphonySdlc.ps1`](scripts/Invoke-PolyphonySdlc.ps1) wrapper) and the [`docs/onboarding-guide.md`](docs/onboarding-guide.md) on-ramp. Today, a new user who follows the guide reaches §9 "First Run" and sees a bare `conductor run apex-driver@polyphony …` invocation with no mention of the wrapper script or the sibling-worktree convention that apex-driver assumes. They will hand-roll the six `-m` metadata flags, get them wrong, and hit the exact failure mode the wrapper exists to prevent. This Epic makes the wrapper and worktree recipe first-class in the onboarding flow so that fresh-clone → first apex-driver run is a copy-paste path.

## Success Criteria

- Onboarding guide contains a section that walks a new user from a fresh clone to a first `apex-driver` invocation using `Invoke-PolyphonySdlc.ps1`.
- The sibling-worktree convention (`../<repo>-<slug>`) is documented with a copy-pasteable recipe covering `git worktree add`, `.twig` setup (`Copy-Item` + `twig set`), and `twig sync`.
- A parameter reference for `Invoke-PolyphonySdlc.ps1` is present, covering at minimum `-ApexId`, `-Intent`, `-Platform`, `-DryRun`, `-NoDetach`.
- At least one worked example shows `-DryRun` output and explains how to read it.
- The [`polyphony-sdlc`](.copilot/skills/polyphony-sdlc) skill cross-links into the new onboarding section, preserving the skill as the deep reference and the guide as the on-ramp.
- All file paths, parameter names, and CLI invocations cited in the new docs match the actual sources ([`scripts/Invoke-PolyphonySdlc.ps1`](scripts/Invoke-PolyphonySdlc.ps1), [`docs/onboarding-guide.md`](docs/onboarding-guide.md)) — no aspirational APIs.

## Scope

### In Scope

- New content in [`docs/onboarding-guide.md`](docs/onboarding-guide.md) covering the worktree recipe, the wrapper walkthrough, parameter reference, and a worked dry-run example.
- A back-link edit in the [`polyphony-sdlc`](.copilot/skills/polyphony-sdlc) skill pointing at the new onboarding section.
- Light edits to §9 "First Run" and the table of contents to integrate the new section, without restructuring the rest of the guide.

### Out of Scope

- Code changes to [`scripts/Invoke-PolyphonySdlc.ps1`](scripts/Invoke-PolyphonySdlc.ps1) or any other source.
- Restructuring the rest of the onboarding guide (sections 1–8, 10).
- New skill files — the existing `polyphony-sdlc` skill remains the deep reference.
- Documenting `conductor` itself or other apex drivers.

## Child Issues

- **task-1** — Document the sibling-worktree bootstrap recipe in the onboarding guide. Establishes the convention and the copy-pasteable setup commands the wrapper assumes.
- **task-2** — Document `Invoke-PolyphonySdlc.ps1` walkthrough, parameter reference, and worked dry-run example. The on-ramp from "worktree exists" to "first apex-driver run".
- **task-3** — Cross-link the `polyphony-sdlc` skill to the new onboarding section. Preserves the skill-as-reference / guide-as-on-ramp split.

## Risks

- **Drift between wrapper and docs** — if [`scripts/Invoke-PolyphonySdlc.ps1`](scripts/Invoke-PolyphonySdlc.ps1) parameters change, the docs go stale. Mitigation: cite the script path and rely on `Get-Help` as the authoritative source; keep the parameter reference minimal (the five required by the AC).
- **Dry-run output drift** — the worked example will show real output that may change. Mitigation: keep the example focused on the structural shape (six `-m` flags, detach behavior) rather than line-for-line transcription.
