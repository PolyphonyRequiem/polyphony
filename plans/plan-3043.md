# Plan — Document `Invoke-PolyphonySdlc.ps1` and worktree convention in onboarding-guide.md

## Strategic Objective

Close the on-ramp gap between [docs/onboarding-guide.md](docs/onboarding-guide.md) and the canonical apex-driver invocation flow shipped in PR #155. New users today reach the end of §9 "First Run" and have no guided path to the supported wrapper ([scripts/Invoke-PolyphonySdlc.ps1](scripts/Invoke-PolyphonySdlc.ps1)) or the sibling-worktree convention apex-driver assumes. The result is hand-rolled `conductor run apex-driver@polyphony` invocations with wrong `-m` metadata — exactly the failure mode the wrapper was built to prevent. This Epic delivers a documented, copy-pasteable path from fresh clone to first apex-driver run, and aligns the [.github/skills/polyphony-sdlc](.github/skills/polyphony-sdlc) skill so the guide is the on-ramp and the skill remains the deep reference.

## Success Criteria

- [docs/onboarding-guide.md](docs/onboarding-guide.md) contains a new section that walks a new user from fresh clone to first `apex-driver` invocation via `Invoke-PolyphonySdlc.ps1`.
- The `../polyphony-<id>` worktree convention is documented with a copy-pasteable recipe (`git worktree add -b sdlc/<id> ../polyphony-<id> main` → `.twig` setup → `twig set` → `twig sync`).
- A parameter reference for `Invoke-PolyphonySdlc.ps1` is present and at minimum covers `-ApexId`, `-Intent`, `-Platform`, `-DryRun`, `-NoDetach`.
- At least one worked example shows `-DryRun` output and explains how to read it (the six `-m` metadata flags, resolved org/project, detected platform).
- The `polyphony-sdlc` skill at [.github/skills/polyphony-sdlc](.github/skills/polyphony-sdlc) cross-links into the new onboarding section so the skill stays the deep reference and the guide stays the on-ramp.
- No code changes; only Markdown under `docs/` and the skill directory.

## Scope

### In Scope

- New section in [docs/onboarding-guide.md](docs/onboarding-guide.md) (logically after §9 "First Run", or as a sub-section of it) covering worktree setup, the wrapper script, parameter reference, and a dry-run worked example.
- Cross-link edits to the `polyphony-sdlc` skill file(s) under [.github/skills/polyphony-sdlc](.github/skills/polyphony-sdlc) pointing into the new onboarding section.
- Updating the guide's Table of Contents to include the new section.

### Out of Scope

- Any code changes to `Invoke-PolyphonySdlc.ps1` or apex-driver itself.
- Restructuring the rest of [docs/onboarding-guide.md](docs/onboarding-guide.md) (§§1–10 stay as-is).
- Creating new skill files; the existing `polyphony-sdlc` skill remains the only deep reference.
- Documenting non-apex-driver workflows (e.g., `actionable.yaml`).

## Child Issues

- task-1 — Add worktree-convention recipe to onboarding-guide.md — establishes the prerequisite environment apex-driver requires, before any wrapper invocation.
- task-2 — Add `Invoke-PolyphonySdlc.ps1` section + dry-run worked example — delivers the wrapper walkthrough, parameter reference, and example output that satisfy AC #1, #3, and #4.
- task-3 — Cross-link `polyphony-sdlc` skill into the new onboarding section — closes the loop so the skill defers to the guide as the on-ramp (AC #5).

## Risks

- **Drift between wrapper help and the doc.** The parameter reference must match the param block at [scripts/Invoke-PolyphonySdlc.ps1](scripts/Invoke-PolyphonySdlc.ps1) lines 67–95; if the script changes, the doc rots silently. Mitigation: keep the reference minimal and link to `Get-Help`.
- **Worktree recipe assumptions.** The convention `<parent>/<repo>-<id>` is enforced softly by the wrapper (regex at line 134); a recipe that picks a different name silently falls back to the worktree itself for the source repo. The doc must show the recipe that matches the regex.
- **Skill ↔ guide ordering.** The skill's existing bootstrap recipe is the current source of truth; merging without coordinating risks two divergent recipes. Mitigation: task-3 explicitly reconciles them.
