# Epic Plan — Research-augmented planning cycle (#3071)

## Strategic Objective

Give every architect agent in polyphony's planning loop the ability to **stop, learn, and resume**: pause mid-plan, dispatch a structured `research_needs` request to a sub-workflow, and resume the next iteration with synthesized findings already in context. Replace today's closed-loop planning (where the architect can only think with whatever was in its prompt) with an open loop that can pull in prior art, verify assumptions against authoritative sources, and accumulate a curated, Obsidian-friendly research wiki in a sibling repo for future runs to discover.

This Epic is also the **dogfood vehicle** for polyphony's first end-to-end run of the full architect → review → plan loop on a non-trivial design problem.

## Success Criteria

- An architect at any planning depth can declare structured `research_needs` and receive back a synthesized findings document before its next iteration.
- A fresh repo opts in via the `research:` block in [`.polyphony-config/profile.yaml`](.polyphony-config/profile.yaml) alone — no code changes required.
- Cross-platform proven in CI or harness: at least one combination of (source platform × research platform) other than `github↔github` works end-to-end.
- Kept articles in the sibling research repo are discoverable via `INDEX.md` and follow the Johnny-Decimal numbering scheme.
- Scratch articles under `research/scratch/<apex-id>/` on the feature branch are pruned cleanly at apex close (verified end-to-end).
- Citations include source URL, capture date, and a freshness signal.
- Three-tier escalation (`research-assistant` → `researcher` → `deep-researcher`) is observable in run telemetry; deep-researcher only fires when the researcher judges the assistant's findings insufficient.
- Research output loops back to the requesting architect's next iteration **without** passing through a plan-review gate.

## Scope

### In Scope

- A `research:` config block in [`.polyphony-config/profile.yaml`](.polyphony-config/profile.yaml) (schema, validation, defaults).
- A cross-platform storage abstraction over the sibling research repo, reusing polyphony's existing platform-router pattern (so source-on-ADO + research-on-GitHub, and the inverse, both work).
- A reusable **research sub-workflow** invokable by any architect at any depth.
- Three agent personas: `research-assistant` (lightweight, archive-only), `researcher` (orchestrator + sufficiency judge), `deep-researcher` (extended tools: web search, GitHub search, MCP-provided integrations).
- An **archivist** pass at the end of every research cycle that decides keep / discard / expand based on relevance to domain, codebase, technology stacks, ecosystem, and linkability.
- Hybrid storage: scratch on the feature branch under `research/scratch/<apex-id>/` (auto-pruned at apex close); kept articles graduated to the sibling research repo as Obsidian-friendly markdown with Johnny-Decimal layout and academic-style citations.
- Architect integration: a `research_needs` output channel and routing that loops research findings directly back into the next architect iteration, bypassing review.
- Wiki polish: Johnny-Decimal numbering, Obsidian wiki-link / frontmatter compatibility, citation rigor (URL + capture date + freshness), and `INDEX.md` maintenance.

### Out of Scope

- Migrating existing in-tree docs into the research wiki.
- Cross-repo research federation (one wiki feeding multiple source repos).
- Human-in-the-loop research authoring UI.
- Replacing or rewriting the existing architect → review → plan workflow shape (research is *additive*; it slots in as a sub-workflow invoked from the architect leg).

## Child Issues

The Epic is decomposed into six Issues that map 1:1 to the phased decomposition the user proposed in the work-item description. Each Issue is sized to fit roughly one merge group so the whole Epic exercises polyphony's wave logic. Order encodes hard dependencies; PG groupings allow some downstream parallelism.

- `issue-1` — **Research storage abstraction + per-repo `research:` config** (PG `pg-foundation`)
- `issue-2` — **Research sub-workflow + `research-assistant` agent** (PG `pg-research-core`, depends on `issue-1`)
- `issue-3` — **`researcher` orchestrator + `deep-researcher` escalation** (PG `pg-research-core`, depends on `issue-2`)
- `issue-4` — **Archivist + curation pass (sibling-repo write)** (PG `pg-curation`, depends on `issue-1`, `issue-3`)
- `issue-5` — **Architect integration: `research_needs` + review-bypass loop-back** (PG `pg-integration`, depends on `issue-2`)
- `issue-6` — **Wiki polish: Johnny-Decimal, Obsidian compat, citation rigor, `INDEX.md`** (PG `pg-polish`, depends on `issue-4`)

## Risks

- **Recursion trap during planning of this Epic itself.** The research subsystem doesn't exist yet, so architects planning Epic #3071 and its children must NOT invoke a research sub-workflow — that's recursion that can't unwind. Guidance is captured directly in the Epic description and must be honored by downstream planners.
- **Cross-platform abstraction surface.** Source-on-ADO + research-on-GitHub (and the inverse) is a hard requirement, not a future. The platform-router pattern must be reused faithfully; greenfield abstractions are a scope risk.
- **Review-bypass invariant.** Looping research output straight back into the architect without a review gate is an intentional design choice. Conductor mechanics around routing must support a clean re-entry into the requesting architect's next iteration without re-triggering the review leg.
- **Scratch-pruning at apex close.** Pruning is end-to-end behavior that depends on apex-close hooks existing or being added. If the close lifecycle doesn't already provide a hook, `issue-2` or `issue-5` may need to add one.
- **Sizing risk on `issue-3`.** The deep-researcher carries the most external-tool surface (web, GitHub, MCP). If extended-tool plumbing turns out larger than one merge group, `issue-3` may need to be split during its own planning pass — that's an Issue-level decision, not an Epic-level one.
