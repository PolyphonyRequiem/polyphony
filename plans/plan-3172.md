# Apex-driver pre-flight refusal on terminal work-item state (audit pass)

## Summary
Ensure [`scripts/Invoke-PolyphonySdlc.ps1`](scripts/Invoke-PolyphonySdlc.ps1) refuses to dispatch when the target work item is in a terminal state (`Done`, `Closed`, `Removed`, `Resolved`), with an explicit operator-visible remediation message naming both `reopen` and `polyphony reset --root-id N`. Per the user's direction, this work is likely already present in the tree — the implementer's job is to audit the existing implementation against the acceptance criteria and gap-fill anything missing.

## Problem / Motivation
Re-dispatching the apex driver against a work item already in a terminal state silently re-runs the SDLC over a closed item, polluting state and producing confusing artifacts (the AB#3165 epic). The driver must fail fast at pre-flight with a clear remediation path so operators recover immediately instead of debugging downstream symptoms.

## Proposed Approach
Audit the Phase 2.5 pre-flight block already present in [`scripts/Invoke-PolyphonySdlc.ps1`](scripts/Invoke-PolyphonySdlc.ps1) (~lines 244–310) and the matching Pester `Describe 'Invoke-PolyphonySdlc — terminal-state pre-flight (AB#3165)'` block in [`scripts/Invoke-PolyphonySdlc.Tests.ps1`](scripts/Invoke-PolyphonySdlc.Tests.ps1) (~line 528+) against the acceptance criteria below. Behaviour required:

1. Calls `twig show $ApexId --output json`, parses the `state` field.
2. If the state is in the terminal set (`Done`, `Closed`, `Removed`, `Resolved`), throws with a remediation message naming BOTH `twig state $ApexId 'To Do'` (reopen) AND `polyphony reset --root-id $ApexId`, plus `-Intent resume` and `-SkipStateCheck` escape hatches.
3. Bypasses the check for `-Intent resume`, `-Intent replan`, and `-SkipStateCheck`.
4. Skips the check (with a warning) when twig is not on PATH or `twig show` fails — never blocks the happy path on probe failure.

If the audit finds the implementation already satisfies every acceptance criterion, the implementer's job reduces to running the existing Pester suite, recording the audit outcome, and moving on. If gaps are found (e.g. wording, missing terminal state, missing bypass test), close them with surgical edits.

## Acceptance Criteria
- [ ] `Invoke-PolyphonySdlc.ps1 -ApexId N -Intent new` against a work item in state `Done`, `Closed`, `Removed`, or `Resolved` exits non-zero **before** any conductor invocation (no `worktree init-apex`, no conductor run).
- [ ] The refusal message names the offending state and references **both** `reopen` (e.g. `twig state $ApexId 'To Do'`) and `polyphony reset --root-id N` as remediation paths.
- [ ] Pester tests in [`scripts/Invoke-PolyphonySdlc.Tests.ps1`](scripts/Invoke-PolyphonySdlc.Tests.ps1) cover each terminal state (`Done`, `Closed`, `Removed`, `Resolved`) plus a non-terminal control (e.g. `To Do` / `Active`).
- [ ] `-Intent resume`, `-Intent replan`, and `-SkipStateCheck` continue to bypass the check (regression-guarded by tests).
- [ ] Behaviour is unchanged for non-terminal states — happy-path tests still pass without modification.
- [ ] Build passes with zero errors and warnings.
- [ ] All existing Pester tests pass; new tests cover the changed behaviour.
