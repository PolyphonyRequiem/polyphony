# ScopeCommands.TagMutationAsync: flush staged tag mutation to ADO

## Summary
Fix a sister-bug to AB#3126 in [src/Polyphony/Commands/ScopeCommands.cs](src/Polyphony/Commands/ScopeCommands.cs): `TagMutationAsync` calls `ITwigClient.PatchFieldsAsync` to mutate `System.Tags` but never calls `SyncAsync` afterward, leaving the change staged in twig's local cache and never pushed to ADO. Downstream processes (e.g. `polyphony validate`) read stale state. Add the post-patch `SyncAsync` flush, mirroring the AB#3126 fix shape, and add a regression test using the established `runner.Invocations` ordering pattern.

## Problem / Motivation
`ScopeCommands.TagMutationAsync` (the implementation behind `polyphony scope add` / `polyphony scope remove` — see lines 158, 177, 180–223) is invoked by workflow steps that hand off to other processes. The staged tag/facet mutation must be durable in ADO before return, otherwise:

- A subsequent `polyphony validate` (or any `IWorkItemRepository` reader) sees stale tags.
- Workflow scripts depending on the just-applied scope facet make incorrect routing decisions.

This is the exact failure mode fixed for `BranchCommands.NextImpl` and `BranchCommands.close-scope` in AB#3126; only the `ScopeCommands` site was missed.

## Proposed Approach
1. In [src/Polyphony/Commands/ScopeCommands.cs](src/Polyphony/Commands/ScopeCommands.cs) `TagMutationAsync`, after the successful `PatchFieldsAsync` call (around line 209) and before the `EmitMutation` success path, add:

   ```csharp
   await twig.SyncAsync(ct).ConfigureAwait(false);
   ```

   Guard the flush behind the existing `changed` branch so we only sync when there is actually staged work (the convention from AB#3126: `count > 0` equivalent).
2. Keep the pre-patch `SyncAsync` at line 182 (it ensures we read fresh tags before deciding to mutate).
3. Add a regression test under `tests/Polyphony.Tests/Commands/` modelled on `BranchCommandsNextImplTests.NextImpl_HappyPath_FlushesStagedTransitionAfterSetState` — assert via `runner.Invocations` that `PatchFieldsAsync` is followed by `SyncAsync` in `TagMutationAsync` for both add and remove paths, and that no second sync occurs when `changed == false` (idempotent no-op case).
4. Convention is already documented in `.github/skills/polyphony-cli-developer/SKILL.md` ("State-mutation durability" section) — no docs update required.

## Acceptance Criteria
- [ ] `ScopeCommands.TagMutationAsync` calls `twig.SyncAsync(ct)` after a successful `PatchFieldsAsync`, only when `changed` is true.
- [ ] Regression test asserts `PatchFieldsAsync` → `SyncAsync` invocation ordering for both `add: true` and `add: false` paths.
- [ ] Regression test asserts no post-mutation `SyncAsync` is issued when the tag set is unchanged (idempotent no-op).
- [ ] Existing `ScopeCommands` tests still pass; failure paths (work-item-not-found, `PatchFieldsAsync` throws) are unchanged.
- [ ] Build passes with zero errors and warnings.
- [ ] All existing tests pass; new tests cover the changed behaviour.
