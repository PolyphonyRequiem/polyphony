# Branch-tip zero-diff short-circuit at apex dispatch

## Summary
Add a pre-flight check at apex dispatch time that compares `feature/{root}` to the configured base branch via `git rev-list --count`. When the count is zero AND the work item's requirements are otherwise satisfied AND the item is non-terminal, route directly to the process-config-defined `item_satisfied` terminal state — emitting a distinct, operator-visible signal rather than a silent success. This closes the AB#3127 reproducer at the apex level so an empty merge group never reaches the scope reviewer.

## Problem / Motivation
When `polyphony` re-dispatches an apex whose `feature/{root}` branch already contains the merged fix, the existing pipeline still walks the planner → impl → merge-group path. The MG arrives empty, the scope reviewer fires the `empty_merge_group_structural_violation` at `scope_revise_cap_gate`, and the run halts on a false positive (AB#3127). The cheapest, highest-signal fix is to detect zero-diff at dispatch time — before any plan/impl/MG sub-workflow is invoked — and exit through a named terminal state.

A silent success would mask future regressions where a real bug produces a no-op impl. The short-circuit must therefore land on a named, observable terminal state sourced from `.polyphony-config/process-config.yaml` (P5 — no hard-coded state names).

## Proposed Approach
Add a new dispatch-preface node to [`.conductor/registry/workflows/apex-item-dispatch.yaml`](.conductor/registry/workflows/apex-item-dispatch.yaml) (and/or its caller [`.conductor/registry/workflows/apex-driver.yaml`](.conductor/registry/workflows/apex-driver.yaml) at the apex entry point) that runs **before** the per-wave routing decision sends the item to a planner / actionable / implement-merge-group / feature-pr sub-workflow.

The new node mirrors the pattern already used by `promote_feature_to_main` in [`apex-driver.yaml`](.conductor/registry/workflows/apex-driver.yaml) (lines ~1168–1206):

1. `git fetch --quiet origin {base} feature/{apex_id}`
2. `git rev-list --count origin/{base}..origin/feature/{apex_id}` (treat exit-code != 0 / missing branch as "not zero-diff" — fall through to normal dispatch).
3. Combine with a requirements-satisfied probe (existing edge/requirements check used by the dispatcher — read, don't re-derive).
4. If `count == 0` AND requirements satisfied AND item is non-terminal: route to a new node that transitions the item to the configured `item_satisfied` state (look up via the `states[<type>].item_satisfied` mapping in [`.polyphony-config/process-config.yaml`](.polyphony-config/process-config.yaml) — already used elsewhere; do NOT hard-code `Done`).
5. Emit a distinct telemetry / log line (e.g. `apex_dispatch.zero_diff_short_circuit`) and a unique terminal route name (e.g. `terminal_apex_already_satisfied`) so an operator scanning runs can distinguish this exit from a normal post-merge close.
6. Otherwise fall through to today's dispatch routing — existing waves and sub-workflows are untouched.

The `{base}` value must come from process-config (typically `main`) — same plumbing the `promote_feature_to_main` site uses today, with the noted caveat in its comment about per-apex target branches becoming a second update site.

Harness coverage lands under [`tests/harness/scenarios/`](tests/harness/scenarios/) with at least one scenario that:
- Seeds an apex whose `feature/{root}` is identical to `main`.
- Runs the dispatcher.
- Asserts the run exits at the new terminal state without invoking the planner, impl, or MG sub-workflow (verifiable via the FakeProvider script log / shim recordings).
- Replays the AB#3127 reproducer (re-dispatch on a root whose `feature/N` already contains the merged fix) and asserts the scope reviewer is never reached.

This Issue is small and focused — a single dispatch-preface node, the terminal-state lookup, and harness coverage all fit within one PG.

## Acceptance Criteria
- [ ] Apex dispatch computes the zero-diff condition against the configured base before invoking any planner / impl / MG sub-workflow.
- [ ] When the condition holds and requirements are satisfied, dispatch routes to a process-config-defined terminal state with operator-visible signal — never a silent success.
- [ ] No state name is hard-coded; the terminal state is sourced from [`.polyphony-config/process-config.yaml`](.polyphony-config/process-config.yaml) via the existing `states[<type>].item_satisfied` mapping (P5).
- [ ] AB#3127's reproducer (re-dispatch on a root whose `feature/N` already contains the merged fix) terminates cleanly and does not reach the scope reviewer.
- [ ] A harness scenario under [`tests/harness/scenarios/`](tests/harness/scenarios/) exercises the zero-diff branch end-to-end and asserts no planner / impl / MG sub-workflow runs.
- [ ] When `feature/{root}` is missing or the rev-list call fails, dispatch falls through to today's routing (no behavioral regression for the non-zero-diff path).
- [ ] Build passes with zero errors and warnings.
- [ ] All existing tests pass; new tests cover the changed behavior (zero-diff hit, missing-branch fall-through, non-zero-diff fall-through).
