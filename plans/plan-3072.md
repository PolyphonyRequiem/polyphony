# Issue Plan — Research storage abstraction + per-repo `research:` config (#3072)

## Summary

Lay the foundation for the research-augmented planning Epic ([#3071](#)) by introducing two things and nothing else: (1) a `research:` config block in [`.polyphony-config/profile.yaml`](.polyphony-config/profile.yaml) with schema, validation, defaults, and round-trip parsing; and (2) a minimal cross-platform storage abstraction (read / write / list) over the sibling research repo, selected at polyphony's existing platform-router seam so source-on-ADO + research-on-GitHub (and the inverse) both work. No agents, no sub-workflows, no archivist — just the config surface and the read/write seam everything else will sit on.

## Problem / Motivation

Every later Issue under [#3071](#) (`research-assistant`, `researcher`, archivist, architect integration, wiki polish) needs two pre-existing facilities: a place to declare *where* the sibling research repo lives + how to authenticate to it, and a uniform API for reading and writing markdown files there regardless of whether that repo is on GitHub or ADO. Without this Issue:

- Downstream Issues each invent ad-hoc config keys → schema drift across the Epic.
- Downstream Issues each call platform SDKs directly → the cross-platform proof criterion (Epic success criterion: at least one non-`github↔github` combo end-to-end) is impossible to factor out.
- The platform-router invariant ([`docs/glossary.md`](docs/glossary.md) §Platform sub-workflow; [`feature-pr.yaml`](.conductor/registry/workflows/feature-pr.yaml) `pr_platform_router`) gets bypassed inside the research leg, fragmenting the pattern.

This is foundation-shaped work: one cohesive seam, no behavioural surface yet, fully unit-testable with mocks at the platform-router boundary.

## Proposed Approach

Two Tasks, one PG (`pg-foundation`), strict ordering. Both Tasks live entirely within [`src/Polyphony/`](src/Polyphony/) and [`tests/Polyphony.Tests/`](tests/Polyphony.Tests/) — no workflow YAML or agent prompts in this Issue.

**Task 1 — `research:` config block.** Extend polyphony's configuration layer to load, validate, and round-trip a `research:` section under [`.polyphony-config/profile.yaml`](.polyphony-config/profile.yaml). Mirror the established loader/validator pair pattern used by [`src/Polyphony/Policy/PolicyLoader.cs`](src/Polyphony/Policy/PolicyLoader.cs) + [`src/Polyphony/Policy/PolicyConfig.cs`](src/Polyphony/Policy/PolicyConfig.cs) and [`src/Polyphony/Configuration/ConfigValidator.cs`](src/Polyphony/Configuration/ConfigValidator.cs). Schema captures: archive `repo` (owner/name), `branch` (default `main`), `platform` (`github` | `ado`, default `github`), `auth` reference (env var name or named credential), and a `paths` block (root path inside the repo, scratch path inside the source repo). Sensible defaults documented inline and in [`docs/`](docs/). YamlDotNet parsing; sealed records; AOT-safe (per [`Directory.Build.props`](Directory.Build.props) conventions).

**Task 2 — Storage abstraction + platform legs.** Define a minimal `IResearchArchive` (or equivalent — exact name/shape decided at implementation time) exposing exactly three operations: `ReadAsync(relativePath)`, `WriteAsync(relativePath, content, commitMessage)`, `ListAsync(relativePathPrefix)`. Two implementations behind it — one GitHub, one ADO — selected by reading `research.platform` from the Task-1 config, mirroring the dispatch shape of [`pr_platform_router`](.conductor/registry/workflows/feature-pr.yaml) but at the C# seam (the workflow-layer router stays for PR lifecycle work; this Issue's seam is in code because no workflow exists yet to invoke it — see open question below). Both legs are unit-tested by mocking the underlying platform clients at the boundary; no live network calls in this Issue. Cross-platform end-to-end *write* proof is explicitly deferred to `issue-4` per the Epic plan.

Decomposition into two Tasks (rather than direct implementation) is deliberate: Task 1 is pure config-surface work reviewable in isolation, and Task 2 cannot be designed until the schema it reads is fixed. Splitting keeps each PR group focused (~300–600 LoC each, well under the 2000 LoC ceiling) and makes the schema review the highest-leverage gate.

## Acceptance Criteria

- [ ] `research:` block parses, validates (clear error messages on missing/invalid fields), and round-trips through profile loading; defaults documented.
- [ ] `IResearchArchive` (or equivalent) exposes `ReadAsync` / `WriteAsync` / `ListAsync` and selects a GitHub or ADO implementation based on `research.platform`.
- [ ] Unit tests cover both platform legs of the abstraction with the underlying platform clients mocked at the seam — no live network.
- [ ] Opting in (and switching platform) requires editing only [`.polyphony-config/profile.yaml`](.polyphony-config/profile.yaml); no code changes in consumer repos.
- [ ] `dotnet build` passes with zero errors and warnings.
- [ ] `dotnet test` passes; new tests cover both Task-1 schema validation paths and Task-2 platform-leg dispatch + per-leg behaviour.
- [ ] Documentation: `research:` schema reference added under [`docs/`](docs/) (sibling to [`docs/polyphony-process-config-schema.md`](docs/polyphony-process-config-schema.md)).

## Child Tasks

- `task-2` — Storage abstraction + GitHub/ADO platform legs + mocked unit tests (PG `pg-foundation`, depends on `task-1`)
