# Epic Plan ‚Äî Research-augmented planning cycle (#3106)

## Strategic Objective

Give any polyphony architect ‚Äî at any depth in the planning tree ‚Äî the ability to **stop, learn, and resume** by invoking a research sub-workflow whose synthesized findings flow back into the next architect iteration as additional context. Accumulated knowledge is curated into a per-repo, Obsidian-friendly sibling wiki so future planning runs can discover prior art instead of re-deriving it.

This breaks the current closed-loop architect cycle (architects can only think with the prompt context they were given) and replaces guesswork / silent-going-wrong with citation-backed, discoverable evidence.

## Success Criteria

These mirror the **Minimum** acceptance bar in the apex (relaxed for first dogfood 2026-05-12). Stretch items are explicitly carried as separate child issues so they can be dropped without de-scoping the Epic.

- An architect at any planning level can declare structured `research_needs` in its output envelope and receive back a synthesized findings document **before** its next iteration runs.
- A fresh repo can opt in to research entirely via a `research:` block in `.polyphony-config/profile.yaml` ‚Äî **no code changes** required to onboard.
- Same-platform end-to-end path is proven: github‚Üîgithub source/research repos, with the research sub-workflow producing a kept article in the sibling repo that the next architect iteration can discover.
- Every citation in a synthesized findings document carries at minimum: source URL and capture date.
- Architect‚Üíresearch‚Üíarchitect loop bypasses the plan-review gate (research is context, not commitment) and this is verified by a passing harness scenario.

## Scope

### In Scope

- New per-repo config schema: `research:` block in `.polyphony-config/profile.yaml` (archive repo identity, branch, auth profile).
- A storage abstraction over the kept-archive sibling repo that reuses polyphony's existing platform-router pattern (so swapping ADO‚ÜîGitHub later is a config change, not a rewrite) ‚Äî even though only github‚Üîgithub is required to land.
- A `research/` sub-workflow with three tiers of agents ‚Äî `research-assistant` (archive lookup only), `researcher` (sufficiency judge / escalation orchestrator), `deep-researcher` (extended tools: web, GitHub search, MCP-provided integrations).
- An `archivist` agent that runs at the end of every research pass and decides keep / discard / expand for each scratch article.
- Scratch storage on the feature branch under `research/scratch/<apex>/‚Ä¶` with auto-prune at apex close.
- Architect agent integration: emit `research_needs` in output envelope, route to research sub-workflow, loop the synthesized findings back into the next architect iteration **without** going through plan-review.
- Harness scenario(s) that exercise the full architect‚Üíresearch‚Üíarchitect loop with FakeProvider scripts.

### Out of Scope

- Migrating existing in-tree docs (e.g. `docs/decisions/`, ADRs) into the research wiki.
- Cross-repo research federation (one wiki feeding multiple source repos).
- Human-in-the-loop research authoring UI.
- Cross-platform proof beyond github‚Üîgithub (carried as a Stretch child issue but may be deferred to a follow-on Epic).
- Freshness signals beyond capture date (Stretch).
- Full Johnny-Decimal `INDEX.md` maintenance (Stretch ‚Äî minimum landing path can write kept articles to a flat or simple layout).

## Child Issues

The six phases in the apex's suggested decomposition map cleanly to six child Issues. Each fits one merge group (one PR group). Phases 1‚Äì5 are required to clear the Minimum bar; Phase 6 carries Stretch acceptance items and may be deferred without invalidating the Epic.

- **issue-1 ‚Äî Per-repo `research:` config + sibling-repo storage abstraction** (pg-foundation): schema, validation, cross-platform-capable remote abstraction (only github‚Üîgithub exercised; ADO path stubbed but unblocked).
- **issue-2 ‚Äî Research sub-workflow skeleton + `research-assistant` agent** (pg-research-loop): the workflow YAML, FakeProvider harness scaffolding, archive-lookup-only first pass.
- **issue-3 ‚Äî `researcher` sufficiency check + `deep-researcher` escalation** (pg-research-loop): adds the two upper tiers and the escalation routing.
- **issue-4 ‚Äî `archivist` curation pass + sibling-repo write** (pg-archivist): keep / discard / expand decision; write kept articles to the configured sibling repo with citation block (URL + capture date).
- **issue-5 ‚Äî Architect integration: `research_needs` emission + review-bypass loop-back** (pg-integration): architect output envelope extension, conductor routing that loops findings back into the next architect iteration, harness scenario covering the full loop.
- **issue-6 ‚Äî Wiki polish: Johnny-Decimal layout, Obsidian wiki-links/frontmatter, `INDEX.md`, freshness signal, scratch-prune verification** (pg-polish): the Stretch acceptance items, deliberately split out so the Minimum bar can land on its own.

## Risks

- **Recursion hazard during planning.** This Epic is the very thing being built ‚Äî architects planning on this Epic must NOT call the research sub-workflow (it doesn't exist yet, and even after it ships, planning *of* the research feature shouldn't depend on it). Captured explicitly in the apex; child issues should re-state it.
- **Cross-platform abstraction over-investment.** Only github‚Üîgithub must work, but the storage layer needs to leave room for ADO‚ÜîGitHub combinations without a rewrite. Risk of either over-engineering (build full cross-platform now) or under-engineering (paint into a corner). Mitigation: reuse existing platform-router pattern; keep ADO path as stub with a clear extension point.
- **Loop-back routing correctness.** Research bypassing the review gate is a deliberate but novel routing pattern. If the architect re-entry doesn't correctly carry the synthesized findings into context, the whole feature is invisible. Mitigation: harness scenario in issue-5 is the contract test for this.
- **Wave-logic stress.** This Epic is also the dogfood vehicle for exercising polyphony's full architect‚Üíreview‚Üíplan loop on a non-trivial design problem. Issue sizing should respect one-merge-group-per-issue so the wave logic actually gets exercised, not bypassed.
