# Re-run idempotency: polyphony state pollution between apex runs

## Strategic Objective

Make polyphony's apex-driver re-dispatch deterministic and useful. Today, re-running `Invoke-PolyphonySdlc.ps1 -ApexId N -Intent new` against a previously-processed work item deterministically loops to `scope_revise_cap_gate` with a false-positive `empty_merge_group_structural_violation` — even after aggressive manual cleanup of branches, worktrees, and per-root state under `<git-common-dir>/polyphony/{root_id}/`. The root cause is a **bug class**, not a single bug: polyphony's outputs (ADO state, ADO tags, branch tips, per-root state) become inputs to the next run on the same root, with no operator-facing way to scrub them and no driver-side refusal to dispatch on terminal states.

This Epic closes the work-item-state pollution surface, complementing AB#3085 (worktree-hijack pollution) and AB#3157 (stale-ref pollution). It blocks any reliable re-dispatch workflow until at least the pre-flight terminal-state refusal and the `polyphony reset` verb land.

## Success Criteria

- A second `apex-driver -ApexId N -Intent new` invocation against an already-completed root produces a deterministic, useful outcome — either a clean short-circuit or an operator-visible refusal — never a 12-cycle false-positive loop at `scope_revise_cap_gate`.
- `polyphony reset --root-id N` exists, is documented, and strips polyphony-authored state (typed via a `PolyphonyTag` DU) from the work item, per-root state dir, branches (local + remote), and per-run worktrees.
- The apex driver refuses to dispatch on `Done`/`Closed`/`Removed` work items by default, with a clear remediation message.
- The branch-tip zero-diff condition short-circuits at apex-dispatch time, before any MG work — so an empty MG never reaches the reviewer.
- A `PolyphonyTag` discriminated union is the single source of truth for what tags polyphony owns vs. what the operator owns.

## Scope

### In Scope

- A new `polyphony reset --root-id N` CLI verb (with confirmation gate and `--force`) that scrubs all polyphony-authored state for a given root: ADO tags (state/intent/facet per the typed DU), polyphony-authored ADO comments (archive then clear), per-root state dir under `<git-common-dir>/polyphony/N/`, per-root branches `feature/N`, `plan/N*`, `impl/N-*`, `mg/N_*`, `sdlc/apex/N` (local + remote), and per-run worktrees under `polyphony-runs/apex-N/`.
- A typed `PolyphonyTag` discriminated union (state vs intent vs facet) so the reset verb — and any future tag-aware code — has a structural source of truth for which tags polyphony owns.
- An apex-driver pre-flight refusal in [scripts/Invoke-PolyphonySdlc.ps1](scripts/Invoke-PolyphonySdlc.ps1) for terminal states (`Done`, `Closed`, `Removed`), with a Pester test.
- An apex-driver pre-flight tag check that surfaces a "previously processed" warning/decision gate when any `polyphony:*` state tag is present (or auto-routes `intent=new` to `intent=resume` semantics).
- A branch-tip zero-diff short-circuit at apex-dispatch time: if `git rev-list --count {base}..feature/{root}` is zero AND requirements are otherwise satisfied, route to `terminal_apex_already_satisfied` (or equivalent existing state) before any plan/impl/MG work.
- A harness scenario asserting the idempotency invariant: a second apex-driver invocation on the same root either short-circuits cleanly or produces an identical artifact set.

### Out of Scope

- The reviewer direction-asymmetry on zero-commit MGs (genuinely-empty vs. nothing-to-do vs. already-merged-forward) — **split out to AB#3166** as defense-in-depth. This Epic's strategy is to never reach the reviewer with an empty MG; AB#3166 is the orthogonal hardening.
- Worktree-hijack pollution (AB#3085) and stale-local-ref pollution (AB#3157) — same shape (state-from-elsewhere leaking into an isolated context), tracked separately.
- Generalizing reset to non-apex roots, multi-root reset, or cross-repo reset — keep the verb scoped to a single root id for now.

## Child Issues

- Issue 1 — Apex-driver pre-flight refusal on terminal state — closes the highest-leverage surface (~30 lines + Pester test); ships hand-built before the rest of the fleet.
- Issue 2 — `polyphony reset --root-id N` verb + typed `PolyphonyTag` DU — provides the operator-facing scrub primitive and the structural source of truth that downstream items depend on.
- Issue 3 — Apex-driver pre-flight tag check — uses the `PolyphonyTag` DU from Issue 2 to detect a previously-processed work item and gate or auto-route.
- Issue 4 — Branch-tip zero-diff short-circuit at apex dispatch — closes the AB#3127 reproducer at the apex level so an empty MG never reaches the reviewer.
- Issue 5 — Idempotency harness scenario *(deferred — track but lower-leverage than 1–4)* — a `tests/harness/` scenario that double-dispatches on the same root and asserts a clean outcome.

## Risks

- **Tag-DU scope creep.** The `PolyphonyTag` discriminated union could grow into a sprawling tag-taxonomy refactor. Mitigation: keep the DU minimal — exactly the tags the reset verb needs to recognize today (state/intent/facet variants).
- **Reset verb destructiveness.** Deleting branches and worktrees is irreversible. Mitigation: confirmation gate by default; `--force` opt-in; archive comments before clearing rather than hard-delete; clear `--dry-run` output.
- **Pre-flight refusal blocks legitimate resume.** Operators may want to re-dispatch a `Done` item intentionally. Mitigation: refusal message points to `polyphony reset --root-id N` and to the `intent=resume` flag; not a silent rejection.
- **Zero-diff short-circuit hides real bugs.** A bug that produces a no-op impl could be masked. Mitigation: short-circuit emits a distinct, observable terminal state (`terminal_apex_already_satisfied`-equivalent) — not a silent success.
