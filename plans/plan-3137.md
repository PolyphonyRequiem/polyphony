# Plan — Wiki polish: numbering, frontmatter, citations, INDEX (AB#3137)

## Summary
Harden the kept-article surface produced by the archivist (AB#3135) so accumulated research knowledge is mechanically discoverable and structurally valid: a declared YAML frontmatter schema, Obsidian-compatible wiki-link resolution, citation rigor (URL + capture date minimum), and a deterministic `INDEX.md` regenerator. All checks are structural and CI-runnable — no GUI dependency.

## Problem / Motivation
AB#3135 introduces the first writes of kept articles into the sibling research repo with a Johnny-Decimal layout and frontmatter. Without machine-checkable structural rules on top of that surface, the wiki will silently drift: frontmatter shapes will diverge between archivist runs, `[[wiki-link]]` targets will rot as articles are renumbered, citations will lose provenance, and `INDEX.md` will accumulate non-deterministic ordering noise that pollutes diffs and breaks reproducibility. The next planning run that tries to discover prior art via the archive cannot trust what it finds.

The parent Epic ([plans/plan-3131.md](plans/plan-3131.md), Issue 6) makes these the explicit acceptance gates for the kept-article tier; this Issue delivers the validators and the deterministic regenerator behind them.

## Proposed Approach
Land four cohesive, mostly-independent pieces, each shipping with CI-runnable validation. The common shape is a small utility per concern that walks the kept-article tree in the sibling repo (path supplied by AB#3132's `research:` config + storage abstraction), reports structural violations, and exits non-zero on failure. The deterministic `INDEX.md` regenerator additionally writes output and is byte-stable for a given input set.

- **Frontmatter schema + validator.** Declare the kept-article frontmatter schema in a single source-of-truth file, write a validator that loads each article's YAML frontmatter and checks it against the schema, and wire it into CI.
- **Wiki-link resolver.** Parse every `[[target]]` (including `[[target|alias]]` and `[[target#heading]]` Obsidian variants) in kept articles and verify each target resolves to an existing article in the sibling repo. Honour the Johnny-Decimal numbering convention adopted by AB#3135 so renames-by-renumber are caught.
- **Citation rigor checks.** Enforce that every citation block includes at minimum a source URL and a capture date (ISO-8601). Surface offending articles with file:line.
- **Deterministic `INDEX.md` regeneration.** Replace any nondeterministic ordering / formatting in the index writer with a fully-ordered, stable serialisation; add a CI check that re-running the regenerator produces a byte-identical `INDEX.md`.

All four pieces consume the sibling-repo path through the storage abstraction from AB#3132 (do not hard-code paths). Where AB#3135 already stamped layout/frontmatter/INDEX behaviour, this Issue tightens the contract rather than rewriting it; if AB#3135's emitted shape conflicts with the schema declared here, raise a follow-up bug rather than silently widening the schema.

Validation surface is structural and CI-checkable — no Obsidian client required.

## Acceptance Criteria
- [ ] Kept articles validate against the declared frontmatter schema (single source-of-truth schema file; CI fails on violation).
- [ ] Every `[[wiki-link]]` target in kept articles resolves to an existing article in the sibling repo (CI fails on dangling links).
- [ ] Every citation in kept articles includes URL and capture date (ISO-8601); CI fails on missing fields.
- [ ] `INDEX.md` regenerates deterministically — re-running the regenerator on the same input set produces a byte-identical file (CI asserts this).
- [ ] Build passes with zero errors and warnings.
- [ ] All existing tests pass; new tests cover changed behavior.

## Child Tasks

- task-2 — Wiki-link resolver (Obsidian-compatible) for kept articles
- task-3 — Citation rigor validator (URL + capture-date minimum)
- task-4 — Deterministic `INDEX.md` regenerator + byte-stability check
