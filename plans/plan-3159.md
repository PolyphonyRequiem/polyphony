# F1: lint-pwsh-jinja-bareword (AB#3156 Bug 1)

## Summary
Add a Pester-based lint that scans `command: pwsh` blocks in conductor workflow YAMLs for the bareword Jinja-rendered boolean trap (e.g. `$x = {{ ... | length > 0 }}`), wire it into CI alongside the other lints, and remediate any existing violations not already fixed by PR #354. Modeled on the existing `tests/lint-prose-children.ps1` + `.Tests.ps1` pattern from PR #244.

## Problem / Motivation
In Jinja-templated PowerShell script blocks, an unquoted boolean expression like `$hasTopics = {{ ... | length > 0 }}` renders to `$hasTopics = true`. PowerShell parses bareword `true` as a cmdlet name, which fails to resolve and assigns `$null` — silently routing the workflow down the wrong branch. PR #354 fixed one occurrence in `plan-level.yaml`, but nothing prevents the pattern from being reintroduced or hiding elsewhere in the workflow set. A typed lint converts this convention into mechanical enforcement (parent epic [#3158](https://dev.azure.com/) — Convert prose conventions to typed enforcement).

## Proposed Approach
Follow the [tests/lint-prose-children.ps1](tests/lint-prose-children.ps1) + [tests/lint-prose-children.Tests.ps1](tests/lint-prose-children.Tests.ps1) pattern (PR #244):

1. **New scanner script** — [tests/lint-pwsh-jinja-bareword.ps1](tests/lint-pwsh-jinja-bareword.ps1):
   - Enumerate workflow YAMLs under `.conductor/registry/workflows/*.yaml`.
   - For each step whose `command:` is `pwsh`, walk the script body lines and flag any line matching the regex `=\s*\{\{[^}]+\}\}\s*$` (a PS assignment whose RHS is an unquoted Jinja expression that consumes the rest of the line).
   - Emit a structured violation per finding (file, line, snippet) and exit non-zero on any violations, mirroring the exit/reporting shape of `lint-prose-children.ps1`.
2. **New Pester tests** — [tests/lint-pwsh-jinja-bareword.Tests.ps1](tests/lint-pwsh-jinja-bareword.Tests.ps1):
   - Positive cases (must flag): bareword RHS with various filter chains, with/without trailing whitespace.
   - Negative cases (must NOT flag): properly quoted shape `$x = '{{ ... }}' -eq 'true'`; non-`pwsh` steps; comments; multi-line expressions that aren't bareword-on-RHS.
   - Repo-scan smoke test that asserts the current tree is clean (after remediation in step 4).
3. **CI wiring** — edit [.github/workflows/ci.yml](.github/workflows/ci.yml) to add a new lint step alongside the existing `Lint (...)` steps, invoking the scanner script (and optionally the Pester tests, matching how `lint-prose-children` is wired).
4. **Remediation** — sweep `.conductor/registry/workflows/*.yaml` for any remaining bareword-Jinja assignments beyond PR #354's fix and convert them to the required quoted `-eq 'true'` shape.

## Out of Scope
Other prose-convention classes (F2/F3/F4) are owned by their own sibling work items. Do not touch them.

## Acceptance Criteria
- [ ] `tests/lint-pwsh-jinja-bareword.ps1` exists and detects the banned pattern in `command: pwsh` blocks under `.conductor/registry/workflows/*.yaml`.
- [ ] `tests/lint-pwsh-jinja-bareword.Tests.ps1` exists with positive cases (flags banned shape) and negative cases (allows quoted shape, ignores non-pwsh steps).
- [ ] CI (`.github/workflows/ci.yml`) runs the new lint as a dedicated step; a synthetic violation in a feature branch fails CI.
- [ ] All existing `command: pwsh` blocks in `.conductor/registry/workflows/*.yaml` are clean against the new lint (any violations beyond the PR #354 fix are remediated).
- [ ] Build passes with zero errors and warnings; all existing tests pass; new tests cover the changed behavior.
