# polyphony reset --root-id verb with typed PolyphonyTag DU

## Summary
Add a new CLI verb `polyphony reset --root-id N` that scrubs every polyphony-authored artifact for a single root (ADO tags + comments, per-root state dir, per-root branches local + remote, per-run worktrees) so an operator has a one-shot path to put a work item back into a re-dispatchable shape. A typed `PolyphonyTag` discriminated union becomes the sole source of truth for which tags polyphony owns; the reset verb consumes it, and downstream items (notably the parent epic [#3165](https://dev.azure.com/) Issue 3) will consume it too.

## Problem / Motivation
When an apex run leaves polyphony state behind (tags, comments, state files, branches, worktrees), the only remediation today is hand-curated `git`/`twig`/filesystem surgery. That is slow, error-prone, and the refusal messages emitted by Issue 1's idempotency guard have nowhere to point the operator. We need a deterministic, one-shot scrub that the guard message can reference, and a typed source of truth for which tags polyphony owns so the scrub stays in sync as new tags are added.

The existing tag surface in [src/Polyphony/Tagging/PolyphonyTags.cs](src/Polyphony/Tagging/PolyphonyTags.cs) is a flat `string`-constant bag (`InScope`, `Root`, `Planned`, `FacetsPrefix`). It works, but it does not classify tags by purpose (state vs intent vs facet) and offers no structural way to enumerate "all polyphony-owned tags currently set on a work item" — exactly what reset needs.

## Proposed Approach
1. **Typed `PolyphonyTag` DU** in `src/Polyphony/Tagging/`: a sealed-hierarchy / discriminated-union shape with three variants — `State` (e.g. `Planned`), `Intent` (`Root`, `InScope`), `Facet` (`polyphony:facets=<csv>`). Keep the variant set MINIMAL — only what's already stamped today; do not pre-expand into a speculative taxonomy. Provide a single `TryParse(string raw)` and an `AllOwned(TagSet)` enumerator that yields every parsed `PolyphonyTag` present on a tag set. Existing `PolyphonyTags` constants stay as the canonical raw strings consumed by the DU.
2. **Reset planner (read-only)** in `src/Polyphony/Reset/`: given a `--root-id`, enumerate every artifact reset would touch: matching ADO tags (via `PolyphonyTag.AllOwned`), polyphony-authored ADO comments (filtered by author/marker), state dir `/polyphony/N/`, branches matching `feature/N`, `plan/N*`, `impl/N-*`, `mg/N_*`, `sdlc/apex/N` (local + remote), worktrees under `polyphony-runs/apex-N/`. Returns a structured `ResetPlan` record that serializes to the verb's JSON output contract.
3. **Reset executor + CLI verb** in `src/Polyphony/Commands/ResetCommand.cs`: ConsoleAppFramework verb following the [HealthCommand](src/Polyphony/Commands/HealthCommand.cs) / `[VerbGroup]` / `[VerbResult]` / `PolyphonyJsonContext` pattern. Flags: `--root-id N` (required), `--dry-run` (planner-only, no mutation), `--force` (skip confirmation), default = prompt before destructive ops. Comment archival writes a sidecar JSON under the per-root state dir (or, when state dir is being deleted, under a sibling `polyphony-archive/N-<utc>.json`) BEFORE clearing comments, so the destructive op is always recoverable. Emit JSON to STDOUT consistent with the contract tests in `tests/`.
4. **Documentation** in `README.md` (and/or a new doc under `docs/`): document the verb, the scrub scope, the archive sidecar location, and the remediation pattern that the Issue 1 refusal message will reference.

## Acceptance Criteria
- [ ] `polyphony reset --root-id N` strips ADO tags matching the `PolyphonyTag` DU, archives + clears polyphony-authored comments, deletes per-root state dir, deletes per-root branches local + remote, and removes per-run worktrees.
- [ ] A typed `PolyphonyTag` DU exists with `State`, `Intent`, and `Facet` variants and is the sole source of truth for which tags the reset verb recognizes; existing constants in [src/Polyphony/Tagging/PolyphonyTags.cs](src/Polyphony/Tagging/PolyphonyTags.cs) feed the DU rather than being duplicated.
- [ ] `--dry-run` enumerates every artifact that would be touched without performing any mutation.
- [ ] Confirmation gate prompts before any destructive action by default; `--force` skips the prompt.
- [ ] Verb has a JSON output contract test under `tests/` consistent with existing verbs (`JsonOutputContractTests`).
- [ ] Verb is documented (README or `docs/`) including the remediation pattern referenced by the Issue 1 refusal message.
- [ ] Build passes with zero errors and warnings.
- [ ] All existing tests pass; new tests cover the DU parser, the planner enumeration, and the executor's archive-before-clear ordering.
