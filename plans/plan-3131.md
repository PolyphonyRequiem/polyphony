# Plan — Research-augmented planning cycle (AB#3131)

## Strategic Objective
Give polyphony architects (at any planning depth) the ability to **stop, learn, and resume** during planning by invoking a research sub-workflow that pulls in prior art, verifies assumptions, and surfaces cross-references that were never in the original prompt. Findings flow straight back into the next architect iteration (no review gate — research is context, not commitment), and curated knowledge accumulates in a sibling Obsidian-friendly research wiki for future runs to discover.

This is also the dogfood vehicle for polyphony's first unattended end-to-end run that exercises the full architect → review → plan loop on a non-trivial design problem.

## Success Criteria (Minimum bar — see Epic for Stretch)
- An architect at any planning level can declare structured `research_needs` and receive back a synthesized findings document before its next iteration.
- A fresh repo can opt in via the `research:` block in `.polyphony-config/profile.yaml` alone — no code changes required.
- Same-platform path (github→github) proven end-to-end.
- Citations include source URL and capture date.

## Scope

### In Scope
- `research:` profile schema + validation, leveraging polyphony's existing platform-router for the sibling research repo.
- A research sub-workflow with the locked three-tier agent pattern: `research-assistant` (archive-only, cheap), `researcher` (sufficiency judge / orchestrator), `deep-researcher` (extended tools — web/GitHub/MCP, escalation only).
- An `archivist` curation pass at the end of every research call (keep / discard / expand).
- Hybrid storage: scratch under `research/scratch/<run>/` on the feature branch (auto-pruned at apex close); kept articles graduate to the sibling research repo with Johnny-Decimal layout, frontmatter, wiki-links, and academic-style citations (URL + capture date minimum).
- Architect integration: emit `research_needs`, loop-back routing that bypasses the plan-review gate.
- `INDEX.md` maintenance for discoverability of kept articles.
- Auth model for the sibling research repo: **reuse the platform-router's existing credential resolution** (gh/az). The `research:` schema accepts an optional explicit auth reference but defaults to platform-router resolution — no per-repo secret plumbing required for the common case.

### Out of Scope (this Epic)
- Migrating existing in-tree docs into the research wiki.
- Cross-repo research federation (one wiki feeding multiple source repos).
- Human-in-the-loop research authoring UI.
- Cross-platform proven combinations beyond github→github (demoted to Stretch per Epic AC).
- Freshness signal beyond URL+capture-date (Stretch).
- Explicit verification that scratch pruning runs end-to-end at apex close (Stretch).

## Approach

Decompose along the six locked phases the Epic suggests. Each Issue is sized to fit roughly one merge group so the run exercises wave logic. Phases 1–4 build the subsystem bottom-up (storage → archive-only call → escalation → curation); Phase 5 wires it into the architect; Phase 6 is the wiki-quality polish that makes accumulated knowledge actually discoverable for the next run.

Dependency edges between children are explicit (see Child Issues list below) so the wave planner does not over-parallelise. The real shape is mostly sequential through issue-1 → issue-2, then a small parallel fan-out: issue-3 (escalation tier) and issue-4 (curation pass) can advance in parallel once issue-2 lands, and issue-5 (architect integration) and issue-6 (wiki polish) can advance in parallel once issue-4 lands.

PGs reflect that shape:
- **pg-1**: foundation (issue-1).
- **pg-2**: archive-only loop end-to-end (issue-2, depends on issue-1).
- **pg-3**: escalation + curation in parallel (issue-3 and issue-4, both depend on issue-2). *Disjoint-surface note:* issue-3 modifies the sub-workflow's intra-loop routing (researcher → deep-researcher escalation step + deep-tier agent definition); issue-4 appends a terminal archivist curation step + sibling-repo writer. They touch the same sub-workflow file at structurally separate insertion points (mid-loop vs. terminal) plus disjoint agent files; the expected conflict zone is the sub-workflow YAML's step list, which should be rebased rather than merged if both land in the same wave.
- **pg-4**: integration + wiki polish in parallel (issue-5 depends on issue-3 and issue-4; issue-6 depends on issue-4).

## Resolved Defaults (from user answers)
- **Sibling research repo auth:** reuse the platform-router's existing credential resolution (gh/az). Issue 1's schema must expose an optional explicit auth override but default to reuse — no new secret plumbing in the common path.
- **Per-tier model selection:** defer to polyphony's existing per-agent model resolution; Issue 2/3 do not pin tier-specific models.
- **Deep-researcher escalation cap:** default to 1 escalation per `research_needs` topic (conservative), overridable via `research:` profile.

## Child Issues
- issue-1 — `research:` profile config + storage abstraction (cross-platform remote, auth via platform-router reuse) — *depends on:* none — *done when:* `research:` block parses/validates from `.polyphony-config/profile.yaml` and the storage abstraction round-trips against a sibling github repo using platform-router credentials, with no new secret plumbing.
- issue-2 — Research sub-workflow + `research-assistant` (archive-only first pass) — *depends on:* issue-1 — *done when:* the sub-workflow accepts a `research_needs` payload and returns a synthesized findings document sourced strictly from the archive, end-to-end, callable from a test harness, **and the `research_needs` payload schema is defined in this issue and consumed unchanged by Issue 5**.
- issue-3 — `researcher` sufficiency check + `deep-researcher` escalation (extended tools, default cap = 1) — *depends on:* issue-2 — *done when:* archive-insufficient cases escalate to the deep tier (web/GitHub/MCP), the escalation cap defaults to 1 and is overridable via `research:` profile, and findings cite both archive and live sources.
- issue-4 — `archivist` curation pass + sibling-repo write — *depends on:* issue-2 — *done when:* every research call ends with an archivist keep/discard/expand decision, kept articles land in the sibling research repo with Johnny-Decimal layout and frontmatter, and `INDEX.md` is updated.
- issue-5 — Architect integration: `research_needs` emission + loop-back routing bypassing review — *depends on:* issue-3, issue-4 — *done when:* an architect can emit `research_needs`, the workflow runs the research sub-workflow, the next architect iteration receives the findings without traversing the plan-review gate, **and a harness / dogfood scenario demonstrates the full architect → research → architect loop on a non-trivial prompt without human intervention**.
- issue-6 — Wiki polish: Johnny-Decimal numbering, Obsidian frontmatter / wiki-links, citation rigor (URL + capture date), `INDEX.md` maintenance — *depends on:* issue-4 — *done when:* **kept articles validate against the declared frontmatter schema, every wiki-link target resolves to an existing article in the sibling repo**, citations include URL + capture date, and `INDEX.md` is regenerated deterministically.

## Risks
- **Recursion trap.** The research subsystem itself is being built here; planning of this Epic must NOT invoke a research sub-workflow (it doesn't exist yet). Called out so downstream architects don't try.
- **Cost blow-up at the deep tier.** `deep-researcher` uses extended tools and a stronger model. The escalation gate in Issue 3 defaults to 1 escalation per `research_needs` topic; widening requires explicit profile opt-in.
- **Sibling-repo auth surface.** Source-on-ADO with research-on-GitHub (or vice versa) is Stretch but the abstraction in Issue 1 must not foreclose it. Default reuses platform-router credentials; the schema must accept an optional explicit auth reference for cross-platform cases without breaking the default path.
- **Dogfood instability.** This is also the first unattended end-to-end SDLC run on a non-trivial Epic. Earlier runs (AB#3106/3107) surfaced four distinct workflow bugs; further latent bugs are likely. Architect/implementers should expect to file follow-up bugs (sister of AB#3127/3130) rather than block the run.
