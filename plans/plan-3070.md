## Summary
Verify that every markdown link in [docs/troubleshooting/launcher-gh-auth.md](docs/troubleshooting/launcher-gh-auth.md) and the inbound cross-links added by sibling `issue-2` resolves to a real target. ## Problem / Motivation
The new troubleshooting doc and its inbound cross-links are only useful if their links work. Broken relative paths, typo'd anchors, or dead in-repo references degrade the doc's value the moment it lands. A focused verification pass after the cross-links are in place catches these before they reach users.

## Proposed Approach
1. Wait for parent prerequisite: sibling `issue-2` (cross-links inbound to the new doc) must be merged first.
2. Enumerate every link in [docs/troubleshooting/launcher-gh-auth.md](docs/troubleshooting/launcher-gh-auth.md) and every inbound link added by `issue-2`.
3. Check the repo for an existing markdown-link-check tool/script (search `scripts/`, `package.json`, `.github/workflows/`). If one exists, use it. Otherwise run a one-shot pass with a standard markdown link checker (e.g. `markdown-link-check`) or do documented manual verification.
4. Resolve each relative path against the repo working tree; resolve each anchor against the target file's heading set; check the outbound link to PR #250 is reachable.
5. Fix any breakage in-place (correct path, fix anchor casing, update URL).
6. Record the verification method (tool invocation or manual procedure) in the PR description.
